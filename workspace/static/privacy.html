Privacy Policy

Last updated: 2026-07-09

Applicant is job-search software you run yourself — as a single self-hosted deployment, not a service Applicant operates on your behalf. There is no Applicant-run server that your résumé, applications, or messages pass through. This page describes, honestly and specifically, what the software stores, what leaves your deployment and when, and how you can export or delete your data.

Who is responsible for your data. Because this is self-hosted software, whoever deploys it — you, or whoever administers the instance you're using — controls the database, the encryption keys, and the machine it runs on. If you're using someone else's deployment, direct data requests to that deployment's administrator; they hold the data, not Applicant.

Where your data lives

A deployment is two databases, both local to wherever you (or your administrator) chose to run them:

  • The workspace (this app you're using right now) keeps a SQLite database — chats, memory, documents, notes, calendar, email accounts, saved model endpoints, and similar workspace data.
  • The engine (the job-search automation underneath) keeps a Postgres database — your profile/résumé data, generated résumés and cover letters, job postings it found, applications and their status, and the screening answers you've approved.

Neither database is reachable from outside the deployment's own network by default, and nothing is synced to any Applicant-operated backend — there isn't one.

What's encrypted

  • Workspace secrets — email account passwords, saved signatures, webhook tokens, and saved model-endpoint API keys — are encrypted at rest with a key file that lives only on the machine running the workspace and is never transmitted anywhere.
  • Engine credentials — passwords banked for filling out job applications — are sealed at rest with authenticated encryption in a dedicated credential store, and are never written to logs.

What leaves your deployment, and when

Applicant's job is to interact with the outside world — that's the point of the product. Here's exactly what goes out, and under what condition:

The AI model you connect

Chat, résumé tailoring, job scoring, and screening-answer drafting all run through whichever model endpoint you connect in setup or Settings. If you point Applicant at a model running on your own machine or local network, nothing leaves your network for these steps. If you connect a remote/cloud provider, the relevant content (chat messages, résumé/profile text, or job posting text, depending on the feature) is sent to that provider — the same as using their product directly. A strict "local-only" mode exists that refuses to use any cloud model for these steps at all, so it can never happen silently even if a cloud endpoint is also configured; it's opt-in because most people connect a cloud model.

Finding jobs

Discovery sends your search criteria — titles, locations, and keywords you configured — to job boards and search engines (through a bundled, self-hosted search proxy) to find postings. This is search terms only, not your résumé or profile.

Applying to jobs

Pre-fill automation visits the employer's own application site (their ATS) and submits exactly the materials you reviewed and approved — that's the product working as intended, the same as if you filled out the form yourself.

Notifications you turn on

Discord and email notifications are opt-in. If you configure them, the text of a notification (e.g. "your digest is ready", "3 applications submitted today") goes to that channel. Leave them unconfigured and everything stays in the in-app notification center.

Updating the software

Checking for and installing updates reaches the usual code-hosting sources (the same as any self-hosted app's update process) — no job-search data is part of that traffic.

Error reports you turn on

Crash telemetry is off by default and opt-in (Settings → System → Error telemetry). If you turn it on and configure a destination for it, a crash sends only a sanitized signature — the error type, a scrubbed one-line message, which part of the app, the app version, and a coarse platform string. It never includes your résumé, job data, passwords, API keys, or a file path with your username in it. There is no Applicant-operated collection server and no default third-party vendor: the destination is whatever HTTP endpoint you supply, so it can be your own self-hosted collector. Leave it unconfigured (the default) and no crash report is ever sent. If you've turned on the strict local-only private mode described below, error telemetry is force-disabled regardless of this setting — see docs/private-mode.md in the project source for the technical detail.

What Applicant never does

  • No analytics or third-party tracking scripts — not in the app, not on the marketing site. The one opt-in exception is the crash-telemetry toggle above, which is off by default, sends only a sanitized crash signature, and has no default destination.
  • No advertising, no selling or sharing data with data brokers, no use of your résumé or job-search activity to train anything beyond your own deployment's local memory.
  • No Applicant-operated server your data passes through, because there is no such server.

Your rights: export and delete

Applicant gives you direct, self-service tools for both — no request or waiting period needed, because it's your own deployment.

Export your data

Go to Settings → Account → Your data → Download my data. This produces a zip with your applications (CSV and JSON), your generated documents (including rendered résumé PDFs where available), your profile, and your recent activity — everything an interrupted job search would need to survive a lost deployment.

Delete your data

  • A specific job search (recommended, and the most complete option today): open the search's Settings and use Danger Zone → Delete this search. This permanently purges that search's résumés, applications, discovery sources, banked credentials, and every parsed profile detail tied to it — and verifies nothing was left behind before confirming the delete.
  • Workspace data by category (chats, memory, skills, notes, tasks, documents, gallery, calendar): an administrator can wipe any of these from Settings → System → Danger Zone, one category at a time.
Being honest about a gap: there is currently no single "delete my entire account" button that removes everything at once across both databases in one click. Deleting every job search plus wiping the workspace categories above covers everything the product stores. A very small number of internal audit-log entries not tied to any specific search may not be swept by a per-search delete — these never include your résumé, profile, or application content, only operational metadata. Because this is self-hosted software, the most thorough option is always available to you directly: stop the deployment and delete its database volumes.

How long data is kept

By default, your data is kept until you delete it — there's no automatic expiry. An administrator can optionally configure a retention window after which parsed profile details and screening answers are pruned automatically, even for searches you haven't deleted.

Shared / multi-person deployments

The workspace supports multiple sign-ins on one deployment, and each account's chats, memory, documents, and similar workspace data are kept separate. The job-search engine underneath, however, is built around a single job search owner — so if an instance is shared, its job-search data (résumés, applications, profile) is not partitioned per sign-in the way workspace data is. If you're on a shared instance, ask its administrator how it's set up.
