# Digest email deliverability

The daily digest and every approval/escalation email ride the same channel:
whatever `apprise_urls` you paste into Settings -> Notifications (or the
onboarding wizard's Notifications step). That channel is the one this product
does **not** control end-to-end — once a message leaves the box, whether it
reaches an inbox or a spam folder is decided by the receiving mail provider's
reputation checks, not by anything the engine does. This page is the honest
contract for that path: what is shipped, what is documented-but-unverifiable
from a dev box, and the checklist to run before you rely on it.

## Recommended default: skip email, use Discord or phone push

**Discord webhooks and phone push (ntfy) have no deliverability problem at
all** — there is no DNS reputation, no spam folder, no SPF/DKIM to configure.
A webhook or a push topic either works or it doesn't, and the in-app "Send
test" button tells you which, immediately. For that reason **Discord and/or
ntfy are the recommended channels** in Settings -> Notifications and in the
onboarding wizard: the panel leads with them and marks them "Recommended",
with email positioned as the option that needs the most setup and offers the
weakest delivery guarantee.

Email is still fully supported (and is the required backstop for the
escalation ladder's 15-minute timeout, `apprise_notifier.py`'s
`_EMAIL_TIMEOUT_SECONDS`), but if you turn it on, treat the rest of this
document as required reading, not optional polish.

## SPF / DKIM / DMARC for the SMTP path

If you configure `apprise_urls` with your own SMTP relay (`mailtos://user:
pass@smtp.yourhost.com:587`, the "Other SMTP" example already shown in the
wizard's help card) rather than a big provider's own send API, the receiving
mail server checks three things against the **domain in your From address**
before it decides whether to trust the message:

1. **SPF (Sender Policy Framework)** — a DNS TXT record on your sending
   domain that lists which mail servers are allowed to send as you. Example,
   for a domain sending only through one SMTP relay:

   ```
   yourdomain.com.  TXT  "v=spf1 mx a:smtp.yourhost.com ~all"
   ```

   `~all` ("softfail") is the safe default while you're setting this up;
   tighten to `-all` ("hardfail") once you've confirmed nothing else needs to
   send as that domain.

2. **DKIM (DomainKeys Identified Mail)** — your SMTP provider signs each
   outgoing message with a private key, and you publish the matching public
   key as a DNS TXT record so the recipient can verify the signature wasn't
   tampered with in transit:

   ```
   selector._domainkey.yourdomain.com.  TXT  "v=DKIM1; k=rsa; p=<public-key>"
   ```

   The `selector` and key are generated by your SMTP provider (Gmail
   Workspace, Fastmail, a self-hosted Postfix + OpenDKIM, etc.) — this
   product does not sign or verify DKIM itself; it only sends through
   whatever relay you configure. Follow your provider's own DKIM setup page
   for the exact selector/key pair.

3. **DMARC (Domain-based Message Authentication)** — a DNS TXT record that
   tells receiving servers what to do when SPF and/or DKIM fail, and where to
   send aggregate failure reports:

   ```
   _dmarc.yourdomain.com.  TXT  "v=DMARC1; p=quarantine; rua=mailto:you@yourdomain.com"
   ```

   Start at `p=quarantine` (failures land in spam, not bounced outright) and
   only move to `p=reject` once your DMARC reports show SPF/DKIM passing
   consistently.

**If you send "as" a free provider address you don't control the DNS for**
(a personal Gmail address used via an app password, for instance), you
cannot add these records yourself — you are relying entirely on that
provider's own reputation and their SPF/DKIM/DMARC, which is one reason a
self-hosted SMTP relay on a domain you own is the more deliverable long-term
setup, and why Discord/ntfy remain the recommended default in the meantime.

### From / Reply-To (already supported, opt-in via the URL)

The underlying send library ([Apprise](https://github.com/caronc/apprise))
already supports a custom From name/address and a Reply-To address as query
parameters on the same `apprise_urls` value — no code change needed to use
them, just add the params:

```
mailtos://user:pass@smtp.yourhost.com:587?from=Applicant&reply=you@yourdomain.com
```

- `from=` sets the display name that shows up as the sender ("Applicant"
  instead of your raw SMTP username).
- `reply=` sets a Reply-To address distinct from the From address — useful if
  you send from a no-reply-style mailbox but still want replies routed
  somewhere you read.

Neither is required, but a stable, real From name is one more small signal
mail providers weigh when scoring reputation, and a working Reply-To means a
"why did you decline my message" from a human recruiter (rare, but it
happens if you ever forward digest content) has somewhere to land instead of
bouncing off a relay-only address.

### Plain-text alternative — already automatic, not a gap

The digest email body is rendered HTML (`DigestService.render_email` in
`src/applicant/application/services/digest_service.py`); `apprise_notifier.py`
detects that and passes Apprise's `body_format=HTML` hint
(`_looks_like_html`/`_send_real_dispatch`). Apprise's own SMTP plugin already
builds a `multipart/alternative` MIME message whenever it is asked to send
HTML — it derives a plain-text part from the HTML automatically and attaches
both parts (see `apprise/plugins/email/base.py`, the `notify_format ==
NotifyFormat.HTML` branch: it builds a `MIMEMultipart("alternative")` with a
converted plain-text `MIMEText` attached ahead of the HTML `MIMEText`). Point
readers who audit this at that file rather than assuming a gap: there is
nothing in this product's own code to add here — the multipart alternative
already ships as a property of the send library every digest email already
goes through.

## Spam-trigger copy review (current digest template)

Reviewed the shipped template (`DigestService.render_email` and
`_digest_subject`) against the common consumer-spam-filter heuristics:

- **No ALL-CAPS shouting, no excessive punctuation** ("!!!", "$$$") anywhere
  in the subject or body — subjects read like `"Your daily digest — 3 new
  matches (top: Backend Engineer at Acme)"`; the empty-day subject is
  `"Daily digest — no new matches today"`.
- **No classic spam-trigger vocabulary** ("free", "act now", "click here",
  "guarantee", "winner", "cash", "urgent!!!") anywhere in the fixed template
  strings — scraped role titles/companies are the only untrusted text in the
  body, and those are HTML-escaped (`html.escape`) and href-scheme-restricted
  (`_safe_href`, `http(s)://` only) before they reach the message, so neither
  a spammy scraped title nor an injected link can ride along.
- **A real, readable footer** explains what the email is and how to change
  delivery ("Change how — and when — these updates reach you in Settings ->
  Notifications") — most spam filters weigh the *absence* of any
  sender-identifying footer copy more heavily than its exact wording, and the
  digest already carries one.
- **No unsubscribe mechanism beyond the Settings pointer.** The footer *tells*
  the reader where to turn delivery off, but there is no `List-Unsubscribe`
  header and no one-click unsubscribe link in the message itself. This is an
  honest, tracked gap (see Known limitations below), not a silent omission —
  large mailbox providers (Gmail, Yahoo, Outlook) increasingly weight the
  presence of a working `List-Unsubscribe`/`List-Unsubscribe-Post` header
  pair in their bulk-sender reputation scoring, so a self-hosted deployment
  sending at any real volume should expect this to matter more over time.

## Bounce handling: honest limitation, not silent tracking

The engine reports a digest email as delivered when Apprise's `notify()` call
returns `True` — i.e. the configured SMTP relay accepted the message for
relay (`NotificationDeliveryError` on `False`, see
`AppriseNotifier._send_real_dispatch`). That is the full extent of delivery
tracking today:

- **Hard bounces, soft bounces, and spam-folder placement all happen AFTER
  the relay accepts the message** and are invisible to the engine — nothing
  parses a bounce DSN (delivery status notification), reads an SMTP
  provider's webhook, or polls a mailbox for returned mail. A "sent" digest
  that the relay accepted and a real mailbox then silently dropped in spam
  look identical from the engine's point of view: both report as delivered.
- **This is deliberately not claimed as monitored.** In keeping with this
  project's honesty invariants (nothing silently drops, nothing claims a
  check it does not perform — see `docs/backlog/road-to-market.md`'s H-series
  and `CLAUDE.md`'s H1-H5 summary), this document states the limitation
  plainly rather than have the product imply bounce-awareness it does not
  have. If you need real delivery/bounce visibility, use your SMTP
  provider's own dashboard (most, including self-hosted Postfix with a
  bounce-processing add-on, expose one) — that is out of this product's
  scope, by design, for v1.

## Live inbox-placement testing: not runnable from this environment

This guidance was authored and the copy/header/template review above was
done by reading source in a sandboxed dev container with **no egress to any
real mail service** — there is no live mailbox this container can send to
and check inbox-vs-spam placement against. That check has NOT been run as
part of this change and this document does not claim otherwise. It is a
procedure for the operator (who has both a real deployment and a real
mailbox) to run themselves:

1. Configure your SMTP relay with a domain you control, add the SPF/DKIM/
   DMARC records above, and wait for DNS propagation (a few minutes to a few
   hours depending on your registrar's TTL).
2. Send yourself a test through Settings -> Notifications' per-channel
   "Send test" button (email row) — this is the honest, real send path
   (`POST /api/setup/channels/test`), not a simulated one.
3. Run the message through [mail-tester.com](https://www.mail-tester.com/)
   or your provider's own reputation tool (Google's
   [Postmaster Tools](https://postmaster.google.com/) if a chunk of your
   audience is on Gmail) and confirm SPF/DKIM/DMARC all show as aligned/pass.
4. Send a real digest (the "send my digest now" control, already reachable
   from the Today rail and the digest page) to your own inbox and confirm it
   lands in the primary inbox, not spam/promotions, on at least Gmail and
   Outlook/Microsoft 365 (the two providers most consumer job-seekers use).
5. Re-check after any change to the From domain, SMTP relay, or DNS records —
   reputation is domain-specific and a new sending domain starts from zero
   trust with every provider.

## Pre-launch deliverability checklist

- [ ] Chose a channel: Discord/ntfy (recommended, zero deliverability setup)
      or email/SMTP with a domain you control.
- [ ] If email: SPF record published and verified (`dig TXT yourdomain.com`).
- [ ] If email: DKIM record published and verified with your SMTP provider's
      own DKIM checker.
- [ ] If email: DMARC record published (`p=quarantine` to start).
- [ ] If email: `from=`/`reply=` set on the `apprise_urls` value to a real,
      readable sender identity.
- [ ] Ran the per-channel "Send test" in Settings -> Notifications and
      confirmed a real delivery (not just the dry-run/offline lane).
- [ ] Ran a live inbox-placement check (mail-tester.com or equivalent)
      against at least Gmail and Outlook/365.
- [ ] Reviewed the digest template for spam-trigger language after any copy
      change (re-run the review above; the current template already passes).
- [ ] Understood the bounce-handling limitation above and, if you need real
      bounce visibility, wired your SMTP provider's own dashboard/webhook
      outside this product.

## Known limitations (tracked, not hidden)

- No `List-Unsubscribe` / `List-Unsubscribe-Post` header pair is emitted.
- No bounce/DSN parsing or spam-folder-placement feedback loop.
- SPF/DKIM/DMARC records themselves must be configured by the operator on
  their own DNS — this product cannot publish DNS records on your behalf,
  it can only tell you what to publish.
