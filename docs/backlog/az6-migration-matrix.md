# AZ6 — Workspace-Data Migration Matrix (D15/D19 Contract)

## Matrix: what migrates, what never migrates

### Migrates

| Workspace entity | Target system | Mechanism |
|---|---|---|
| `sessions` (chat sessions) | A0 history | direct |
| `documents` (personal docs) | A0 canvas files | direct |
| `notes` (Keep-style notes/checklists) | A0 canvas files | direct |
| `memory_entries` (general memories) | A0 memory | direct |
| `skills` (skill definitions) | A0 memory | direct |
| `job_search_memories` | Engine mind | curation-gated (D19) |

### Never migrates (engine-owned truth — D15)

| Entity | Reason |
|---|---|
| `applications` | Engine-owned truth — `Application` entity (`src/applicant/core/entities/application.py`) carries the lifecycle, §7 state machine, and submission state |
| `generated_documents` | Engine-owned truth — `GeneratedDocument` entity covers resumes, cover letters, screening answers with redline state; the engine owns document generation and approval |
| `provenance` | Engine-owned truth — `LearnedProvenance`/`LineProvenance` are advisory transparency records (FR-MIND-5/-11) generated by the engine's curation pipeline; they are not workspace content |
| `curation` | Engine-owned truth — `CurationLedger`/`CurationService` are the engine's curation pipeline (FR-MIND-11); the curation gate controls what memories/skills enter the engine mind |
| `attributes` | Engine-owned truth — `Attribute`/`AttributeStore` entities (FR-ATTR-*) are per-campaign attribute clouds; values are campaign-scoped and engine-authoritative |

## Design rationale

### Why engine-owned state NEVER moves (D15)

The applicant engine is the single source of truth for its core domain entities. Workspace is a UI/front-door layer that stores ephemeral user data (chats, notes, personal documents). The engine's entities — applications, documents with redlines, provenance records, the curation ledger, and attribute stores — participate in the engine's state machine, business rules, and data integrity constraints. Copying them to A0 would fork the authoritative truth and create inconsistency.

### Why job-search memories go through curation-gated import (D19)

Workspace may contain general memories (preferences, facts) that are safe to copy directly, and job-search-relevant memories that must be filtered by the engine's curation ledger before entering the engine mind. A direct copy would bypass the curation gate (FR-MIND-11), potentially importing irrelevant or low-confidence memories into the engine's decision-making substrate.

### Separate executable migration step

This document defines the CONTRACT. The actual migration executable (dry-run on an isolated data copy, backup-first, per-slice fidelity counts) is a separate, gated deliverable governed by this contract.

## Enforcement

The invariants defined here are enforced by [`tests/unit/test_migration_matrix.py`](/tests/unit/test_migration_matrix.py), which verifies:

- Disjointness: no entity appears in both migrates and never_migrates
- D15 invariant: every engine-owned entity is present in never_migrates and absent from migrates
- D19 invariant: the job-search-memory row uses `curation-gated` mechanism
- Schema completeness: every row has the required fields
