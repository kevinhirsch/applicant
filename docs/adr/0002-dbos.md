# ADR-0002: DBOS Transact for durable execution

**Status:** Accepted (mandated by master spec §3.15 FR-DUR-3; resolved through v4 per §12).

## Context

The engine runs 24/7 and must survive mid-step crashes with no lost long-running work (FR-DUR-1, NFR-247-1): a killed worker must resume from the last completed step. Each application is a long, multi-step flow (navigate, fill field, screenshot, score, generate) with human-in-the-loop approval waits and revision hand-offs. It also needs concurrency caps (sandbox), per-provider LLM rate limiting, and cron-like scheduling (digests, discovery). The deployment is a single self-hosted Proxmox VM where operational weight matters, and the stack already deploys PostgreSQL.

Alternatives considered (§3.15): **Temporal** (self-hosted server is free but operationally heavy — a separate orchestration server); **Inngest** (leans SaaS); **Camunda** (heavy JVM/BPMN); **ZenML** (ML-pipeline only). **Restate** MAY be evaluated later as a lighter alternative.

## Decision

Use **DBOS Transact** (MIT, open-source, free, self-hosted) — a lightweight durable-execution **library** that stores workflow/step state in the **same Postgres we already deploy**, with **no separate orchestration server**. Mapping:

- Each application = a durable **DBOS workflow**; each small idempotent step is individually checkpointed → true mid-step resumption.
- **DBOS durable queues** enforce the sandbox concurrency cap and per-provider LLM rate limiting.
- **DBOS `send`/`recv`** implement the approval gates and revision-loop hand-offs.
- **DBOS scheduling** drives cron-like work (digests, discovery). LangGraph is the reasoning loop *within* steps.

## Consequences

- **Positive:** No extra server to operate (fits the single-VM self-hosted target, §11 "durable execution is lightweight here"); durable state co-resides with application data in one Postgres (§8); MIT-licensed and free; DBOS emits OTel traces/metrics for workflow visibility (FR-OBS-1). Phase 0 exit gate verifies a trivial workflow resumes after a kill.
- **Negative / cost:** Steps must be written **idempotent** and small to get true mid-step resumption; couples durable execution to Postgres; ties the orchestration model to DBOS's primitives (mitigated by the DurableOrchestration **port** — Restate could be swapped behind it).
- **Library, not framework:** runs in-process; no separate deployment artifact.
