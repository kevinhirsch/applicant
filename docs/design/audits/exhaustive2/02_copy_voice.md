# Copy & voice — line-by-line audit (exhaustive pass 2, lens 02)

> **Lens.** Every user-facing string in the Applicant lane — front-door JS surfaces
> (`workspace/static/js/applicant*.js`, `documentLibrary.js` applicant lane,
> `emailLibrary/applicantDigest.js`, `index.html` titles/tooltips) and engine-emitted
> message templates — judged against ONE house voice: **first-person-singular, calm,
> plain, quietly confident, on-your-side.** Never chirpy, never engineering vocabulary,
> never third-person self-reference ("Applicant is…", "the assistant…"), no exclamation
> spam, sentence case throughout.
>
> **Dedup.** Moment-level copy items from `PRODUCT_EXHAUSTIVE_AUDIT.md` / `exhaustive/delight.md`
> are not repeated; already-implemented features (warm greeting/empty state, never-does
> contract, recap + momentum, starter prompts, snapshot preview, global pause, Results)
> are audited only for the wording of their *current* strings.
>
> Format: `N. **Title** — [VALUE · EFFORT] — before → after — file:line`. Ranked, high value first.

## Cross-cutting (fix once, fixes dozens)

1. **Pronoun anarchy: "I" / "we" / "Applicant" / "the assistant" / "your assistant" all speak for the same agent** — [VALUE: high · EFFORT: M] — the same product says "I'll bring anything that needs you right here" (applicantPortal.js:608), "The assistant has a question for you" (applicantPortal.js:177), "Snoozed — we’ll remind you tomorrow" (applicantPortal.js:978), and "What Applicant never does" (applicantPortal.js:593). Pick first-person-singular "I" everywhere the agent speaks; reserve "Applicant" for the product name in chrome (nav, titles), never as the sentence's actor. This one rule rewrites ~40 strings below.
2. **Raw `e.message` shown to users as toast copy** — [VALUE: high · EFFORT: S] — pattern `_toast(e.message || 'Could not …')` surfaces proxy/engine internals ("HTTP 502 Bad Gateway", JSON parse errors) as the agent's voice. before: `_toast(e.message || 'Could not update that')` → after: map through `errText(e)` (plain-language, already exists in applicantCore.js) so users only ever see calm, plain failure copy — applicantPortal.js:963,982,1015,1039,1063,1131,1168,1208,1226 (and the same pattern in other surfaces noted below).
3. **Typography drift: straight vs curly apostrophes in the same view** — [VALUE: low · EFFORT: S] — "Today's roles" (straight, applicantPortal.js:1315) sits above "Loading today’s roles…" (curly, applicantPortal.js:1347); "you're" straight at :606 vs "we’ll" curly at :978. Pick curly (’) everywhere in user-facing copy and sweep the applicant lane once.
4. **Terminology drift: campaign vs search** — [VALUE: high · EFFORT: M] — user-facing strings say "job search" in some places (applicantPortal.js:1308 "Choose which job search…") and "campaign" in others (see applicantCampaignSettings.js / applicantDigest.js / applicantDebug.js items below). "Campaign" is CRM vocabulary; standardize every user-facing instance on "search" ("your search", "this search") and keep `campaign` only in code/API names.
5. **Terminology drift: materials vs documents** — [VALUE: med · EFFORT: S] — the same artifacts are "A tailored document is ready" (applicantPortal.js:181), "Materials approved ✓" (applicantPortal.js:775), and "Review applications" deep-links into a surface called the Library. Standardize on "documents" (the nav noun) for the artifacts and reserve "application" for the whole submission.
