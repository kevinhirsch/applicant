# Applicant — Power Users, Efficiency & Extensibility Audit

**Lens:** what a technical self-hoster / power user *cannot yet do*: drive the product from the
keyboard, from scripts, from the workspace's own agent, from other agents (MCP), from webhooks and
schedules; operate in bulk; get data in and out; run multiple campaigns like a pro.
**Dedup:** the headline items already in `PRODUCT_EXHAUSTIVE_AUDIT.md` + `exhaustive/ux-flows-ia.md`
(#1 hash routing, #8 command palette, #9/#38 shortcut maps) and `exhaustive/product-gaps.md`
(#43 MCP verify, #45 CSV export, #47 LinkedIn import) are **not repeated at headline level** — where
they appear here it is at the specific implementation/anchor level, or because the verified facts
change their scope/effort. Everything else is new territory (tokens, agent loopback, webhooks,
scheduler recipes, CLI, rules, multi-campaign muscle, dev mode).

**Ranked best-ROI first within each tier.** Format: `N. **Title** — [VALUE · EFFORT] — rationale + anchor.`

---

## Tier 1 — The workspace's own power rails, unwired to the engine (highest ROI: the rails exist)

The vendored front-door already ships an industrial power-user substrate — scoped bearer API tokens,
a ~57-tool agent with a generic `app_api` loopback, a full MCP client, an outgoing-webhook manager,
and a cron/event task scheduler. Almost none of it is connected to the applicant engine.

1. **Give the chat agent first-class applicant tools ("pause my search", "approve the Stripe role")** — [VALUE: high · EFFORT: M] — The generic `app_api` loopback already allowlists the whole engine proxy (`workspace/src/tool_implementations.py:2688` `"/api/applicant/"`) and impersonates the calling owner via the internal-token headers (`:2459-2464`), so the agent *technically* can pause search (`applicant_control_routes.py:92` pause-all), approve a digest row (`applicant_email_routes.py:235`), or query postings (`applicant_compare_routes.py:137`) — but there is **no named applicant tool** in the registry (`workspace/src/tool_schemas.py`; tag set `agent_tools.py:29-61` has zero applicant entries) and no system-prompt affordance, so the model must invent raw REST paths against an undocumented surface. Add ~6 wrapped tools (`applicant_status`, `list_pending_actions`, `approve_digest_item`, `decline_digest_item(reason)`, `snooze_action`, `pause_search`/`resume_search`) over the existing proxies. This single change makes chat the product's command line.
2. **Method-fence consequential applicant writes inside `app_api`** — [VALUE: high · EFFORT: S] — The applicant prefix is allowlisted *wholesale* (`tool_implementations.py:2688`); the method+path blocklist (`:2696-2720`) fences cookbook/email writes but nothing under `/api/applicant/` — so the LLM can POST `remote/submit-self` or `remote/authorize-engine-finish` (`applicant_remote_routes.py`) through the loopback. The engine's server-side stop boundary still holds (core-enforced), but defense-in-depth demands the tool layer refuse the final-authorization paths outright, exactly as it already refuses `/api/cookbook/state` overwrites. Never let "the guard is downstream" be the only guard.
3. **Make bearer API tokens actually work on `/api/applicant/*` (owner attribution)** — [VALUE: high · EFFORT: M] — Token verification stamps `request.state.current_user = "api"` — the literal sentinel, not the token's owner (`workspace/app.py:383`; owner kept separately at `:386`). Every applicant proxy gates only on `require_user` (`workspace/src/auth_helpers.py:15-35`; e.g. `applicant_portal_routes.py:95`), which returns `"api"` — so a token call passes auth but operates as a phantom user with no campaigns: `curl -H "Authorization: Bearer …" /api/applicant/portal/pending` "works" and returns nothing useful. No applicant route reads `api_token_owner` (grep: only `app.py` + `webhook_routes.py`). Resolve owner from `api_token_owner` in the applicant proxies (or a shared dependency) and the whole scriptable-API story (CLI #22, palette #43, reports) lights up.
4. **Real token scopes: `applicant:read` / `applicant:act`** — [VALUE: high · EFFORT: M] — Scopes are hard-coded: `DEFAULT_SCOPES = "chat"` with no parameter to set anything else (`workspace/routes/api_token_routes.py:17,71`), and scope is enforced in exactly one place — the public chat endpoint (`workspace/routes/webhook_routes.py:199-201`). Once #3 lands, a token that can *read* the funnel must not implicitly be able to *approve* applications. Add scope choices at mint time + a scope check in the applicant proxies (read vs act; never a submit scope).
5. **Non-admin token minting UI** — [VALUE: med · EFFORT: S] — Every token route is `require_admin` (`api_token_routes.py:25,54,87`) and the only UI is the admin panel (`workspace/static/js/admin.js:1186,1767-1800`). On a multi-user instance the non-admin owner — the person who'd script their own job search — cannot mint a personal token. Per-user tokens scoped to the caller are the norm; keep admin oversight via the existing list.
6. **The engine's MCP server is unreachable in every real deployment — pick its story** — [VALUE: high · EFFORT: M] — The engine mounts a dependency-free MCP tool surface at `/mcp/tools` + `/mcp/tools/call` (`src/applicant/app/routers/mcp.py:196-199`, wired `app/main.py:119`), but: the `api` service is expose-only (`docker/docker-compose.prod.yml:212`), the workspace proxies every functional engine router *except* `mcp` (verified against `workspace/src/applicant_engine.py`), and the image never installs the `mcp` extra (`docker/Dockerfile:151` installs only `--extra browser`), so the SSE transport never mounts (`mcp.py:202-208`, `pyproject.toml:82-83`). Net: a flagship differentiator for exactly this buyer ("drive Applicant from your own agents") is dead code in prod. Either proxy `/mcp` through the front-door behind token auth, or preconnect it internally (#7) — today it is neither.
7. **Preconnect the engine as a built-in MCP server of the workspace agent** — [VALUE: high · EFFORT: S] — The workspace is a full MCP *client* (stdio+sse, `workspace/src/mcp_manager.py:29-51`; users can add servers via `POST /api/mcp/servers`, `workspace/routes/mcp_routes.py:82-104`; the agent itself can via `manage_mcp`), with built-ins for image/memory/rag/email (`workspace/src/builtin_mcp.py:69-74`) — but nothing registers `http://api:8000/mcp`. Adding it as a built-in (gated on the internal token being configured) gives the agent structured, schema'd engine tools instead of `app_api` guesswork, and it composes with #1.
8. **Engine MCP tools accept zero arguments** — [VALUE: med · EFFORT: S] — Every advertised tool has `inputSchema: {"type":"object","properties":{}}` (`mcp.py:69`); `get_applications` returns *all* applications with no campaign/status/company filter (`:141-149`). At real volume that's unusable and token-expensive for any calling agent. Add filter parameters (campaign_id, status, since, q) to the read tools.
9. **Expose a guarded write tier over MCP (never submit)** — [VALUE: med · EFFORT: M] — The call handler default-denies everything but the 5 read tools (`mcp.py:95-108`) — correct for final-submit, but pause/resume, snooze, and decline-with-reason are human-safe operations the HTTP surface already exposes; denying them makes MCP a toy. Add them with the same server-side gates, keeping the submit boundary hard-denied.
10. **Token-gate the MCP endpoints before any exposure** — [VALUE: med · EFFORT: S] — `/mcp/tools` and `/mcp/tools/call` have no auth dependency at all (`mcp.py:56-77,76-109`) — acceptable only while unreachable (#6). Any proxying/preconnect must add bearer/internal-token gating first; do it now so #6/#7 can't ship open.
11. **Document the MCP surface** — [VALUE: med · EFFORT: S] — `/mcp/tools` appears nowhere in `docs/` (`docs/extending.md` covers code-level adapters only; `product-gaps.md #43` asked "verify/finish" — it exists but is invisible). One page: what's exposed, the read-only contract, curl examples, client config (workspace MCP settings + external agent clients), and the guarded-write roadmap.
12. **Bridge engine domain events → outgoing webhooks** — [VALUE: high · EFFORT: M] — The workspace webhook manager (HMAC-signed, SSRF-guarded, `workspace/src/webhook_manager.py:1-70`) fires on chat events only (`:20-25`: `session.created`, `chat.completed`, `chat.message`, `webhook.test`); the engine's `DomainEventBus` carries 6 domain events strictly in-process (`src/applicant/core/events.py:33-64`), and the engine's only "webhook" is the Discord notification URL (`adapters/notification/apprise_notifier.py:170`). Fan engine events (digest.ready, action.needed, application.submitted, run.error, interview.detected) into the webhook manager's event set — self-hosters wire Home Assistant/n8n/Slack in minutes, no polling.
13. **Ship applicant automation recipes on the existing task scheduler** — [VALUE: high · EFFORT: S] — The scheduler is real user-facing automation: cron/daily/weekly/monthly/once + event triggers (`workspace/src/task_scheduler.py:73-160`), executing the *full agent tool loop* owner-scoped (`:1481-1539`), with built-in recipe seeding (`:195-204` — currently an email summarizer). "Every weekday 8am: summarize my pending actions and email me" nearly works today via `app_api`, but there are zero applicant built-ins and zero docs. Seed 2–3 recipes (morning pending-actions brief, weekly funnel report, stale-application nudge) — this is scheduled reports for free.
14. **Applicant events as scheduler triggers** — [VALUE: med · EFFORT: M] — The scheduler supports event-triggered tasks (`task_scheduler.py:200-204`), but only workspace events exist. Once #12 lands, "when a digest arrives, run this prompt" / "when an action needs me, ping ntfy with a summary" become user-composable automations — the recipe system the brief asks for, assembled from parts that already exist.
15. **Document ntfy as a scripting hook** — [VALUE: low · EFFORT: S] — The engine's channels are discord/in_app/email/ntfy (`adapters/notification/`); ntfy topics are trivially consumable by shell scripts (`ntfy subscribe … | while read …`) — a zero-code automation hook for self-hosters that no doc mentions. One paragraph in the setup/notification docs.

## Tier 2 — Programmatic access, data portability, CLI

16. **Entity search/query API (the keystone)** — [VALUE: high · EFFORT: M] — No engine listing endpoint accepts `q`/company/title/status parameters — all are path-scoped and unpaginated (verified across `app/routers/`: campaigns/gallery/pending_actions/digest expose only ids, `include_snoozed`, caps). "Find my Stripe application" is impossible except by eyeballing. Add `?q=&status=&company=&limit=&offset=` to applications/postings listings (+ proxy passthrough). This unblocks the command-palette entity provider (ux-flows #8), the agent tools (#1), MCP filters (#8), and the CLI (#22) simultaneously — one endpoint, four power surfaces.
17. **Owner-scoped applications/tracker export (CSV + JSON)** — [VALUE: high · EFFORT: S] — The engine's only export is admin-gated audit-log JSON (`src/applicant/app/routers/audit.py:59,74`; front-door button `applicantDebug.js:266-290`, admin Debug only). Nothing on Results/Gallery/Portal/Digest exports anything; no CSV exists anywhere in the engine. `product-gaps.md #45` names the need — the concrete shape: `GET /api/applicant/results/export.{csv,json}` (owner-scoped, all campaigns, one row per application with status/company/dates/source/variant) + an Export button on the Results surface (`applicantResults.js` is currently read-only + Refresh). Benefits reporting, spreadsheets, and "my data is portable" trust in one small endpoint.
18. **Attribute-cloud + criteria export/import (JSON round-trip)** — [VALUE: med · EFFORT: S] — `attributes.py`/`criteria.py` are item-CRUD only; there is no bulk GET-as-file and no import. The user's most painstakingly-entered data (the Workday-ready profile) cannot leave or re-enter the system — a reinstall means re-typing onboarding. A versioned JSON export/import (distinct from a DB backup: user-grade, diffable, editable) also enables sharing sanitized profile templates.
19. **Audit-log export filters** — [VALUE: low · EFFORT: S] — `export.json` takes no date-range or state filters (`audit.py:59-74`); unemployment-benefits reporting needs "applications submitted between X and Y." Add `?from=&to=&status=`.
20. **Bulk materials download per application** — [VALUE: low · EFFORT: S] — Variant PDFs download one at a time (`documents.py:134`); there is no "download everything sent for this application" (resume + cover + screening answers + snapshot) as a zip — the artifact a user wants when an interview lands.
21. **Effective-config introspection** — [VALUE: med · EFFORT: S] — `Settings` has ~90 env-driven fields (`src/applicant/app/config.py:148-773`) and zero introspection: no `/config` endpoint, no dump command; `/healthz` reports version+capabilities only (`app/main.py:121-184`). Debugging "why is discovery not running" today means `docker exec` + reading source. A redacted `GET /api/applicant/admin/config` (secrets masked) — and its CLI twin (#22) — is table stakes for a self-hosted product.
22. **`applicantctl` — a thin CLI over the proxy API** — [VALUE: med · EFFORT: M] — No CLI is packaged (no `[project.scripts]` in `pyproject.toml`, no `__main__.py`; `scripts/` is ad-hoc ops). NFR-ZEROCLI-1 (`docs/spec/master-spec.md:268`) mandates that no setup step *requires* the CLI — it does not forbid one, and this buyer expects one. `applicantctl status|pending|approve|decline|pause|resume|export|config` over `/api/applicant/*` with a bearer token (needs #3) is a weekend of work once the API story is fixed; the ops endpoints it would wrap already exist (`applicant_ops_routes.py`; `docs/autonomous-agent-howto.md:89-134` literally teaches them as curl commands today).
23. **Publish an API reference for the proxy surface** — [VALUE: med · EFFORT: S] — The engine's FastAPI OpenAPI is internal-only; the workspace publishes no reference for its ~20 `/api/applicant/*` route families. Power users must read route source to script anything. Generate/commit an OpenAPI subset (or a hand-written `docs/api.md`) for the applicant proxy surface, cross-linked from the token UI.
24. **A public health probe distinguishing UI-up from engine-up** — [VALUE: low · EFFORT: S] — Engine `/healthz` (`app/main.py:121`) is reachable only in-network; self-hosters running uptime monitors against `${APP_PORT}` can only see the front-door. Expose (and document) an unauthenticated `/healthz` on the workspace that includes an engine-reachability bit.

## Tier 3 — Bulk operations & triage efficiency

25. **Bulk decline-with-one-reason** — [VALUE: high · EFFORT: M] — Declining a digest role requires a typed reason *per item* (`workspace/static/js/applicantDigest.js:373-388`), and the engine's only bulk endpoint is pending-action `resolve-bulk` (`src/applicant/app/routers/pending_actions.py:100`) — there is no bulk decline anywhere. "Decline these 8: salary below floor" is the most-repeated power move in daily triage, and the single shared reason is *better* learning signal than eight rushed ones. Needs multi-select (#26) + one engine endpoint.
26. **Multi-select rows (checkboxes) on Portal + digest lists** — [VALUE: high · EFFORT: M] — No list in any applicant surface supports selection; the Portal's only bulk affordance is all-or-nothing "Approve all N" (`applicantPortal.js:864-868,987-1019`). Select-some → approve/decline/snooze is the difference between an inbox and a queue you fight. (Complements quick-wins #21, which asked only for approve/decline-all.)
27. **Bulk snooze / vacation mode** — [VALUE: med · EFFORT: S] — Snooze is single-item (`pending_actions.py:121`; UI `applicantPortal.js:632-645`). "Snooze everything until Monday" — one button + a loop over the existing endpoint (or a bulk variant) — is the polite version of the kill switch when the user travels.
28. **Snooze durations beyond "tomorrow"** — [VALUE: med · EFFORT: S] — The snooze is hard-coded to tomorrow morning (`applicantPortal.js:633` "Hide this until tomorrow"; engine `pending_actions.py:121`). Add 3 days / 1 week / until-date as a parameter — trivially generalizes the existing endpoint.
29. **Snooze on digest rows, not just Portal rows** — [VALUE: med · EFFORT: S] — The digest triage offers approve/decline only (`applicantDigest.js`); a role you want to *think about* must be declined or left pending. Deferring a digest row is a natural third verb the Portal already models.
30. **Approve-all as a REST endpoint** — [VALUE: med · EFFORT: S] — Digest approve-all exists only as chat natural language (`application/services/chat_service.py:1078-1190` `_do_approve_all`); scripts, tokens, and the CLI can't reach it. Promote to `POST /digest/{cid}/approve-all` and have chat call it — one behavior, two entrances.
31. **Browsable application list in the Documents library** — [VALUE: high · EFFORT: M] — The applicant tab of the library is a hand-typed "Application ID…" lookup (`workspace/static/js/documentLibrary.js:1903-1938`) — no list, no filters, no sort, no pagination, and IDs are shown nowhere the user can copy (except Compare, #32). The materials-review surface — the daily consent gate — cannot even enumerate what exists. Render the owner's applications as a filterable list (needs #16 for search).
32. **Expose entity IDs with click-to-copy everywhere** — [VALUE: low · EFFORT: S] — Click-to-copy IDs exist only inside Compare (`applicantCompare.js:237,255-257`), yet Compare's own input requires hand-typed IDs (`:106-110`) and the ops/compare/library surfaces all consume them. Until pickers exist, put a copyable ID on Portal/Gallery/Results rows — the duct tape that makes the current power workflows possible at all.

## Tier 4 — Keyboard & command-palette (the specifics under the known headlines)

33. **The keybind *customization UI* can't target applicant surfaces either** — [VALUE: high · EFFORT: S] — ux-flows #9/#38 flagged the runtime maps; the deeper find is that custom keybinds are a real, server-persisted feature (`workspace/static/js/settings.js:1682-1969` → `/api/auth/settings`) and the applicant surfaces are missing from *all four* layers: `_defaultKeybinds` (`keyboard-shortcuts.js:5-14`), `_WINDOW_TRIGGERS` (`:92-105`), `_toolBtns` (`:249-260`), *and* the settings editor's action list — so even a motivated user cannot self-bind Portal/Activity/Results/Assistant/Archive/Gallery (`index.html:850-862` rail IDs). Adding the six `open_*` actions to the two maps + the editor is one small, closed change.
34. **Keybind default mismatch (bug): sidebar toggle** — [VALUE: low · EFFORT: S] — Runtime default is `ctrl+alt+b` (`keyboard-shortcuts.js:6`) while the settings panel displays `ctrl+b` as the default (`settings.js:1684`) — the panel advertises a binding that doesn't work until re-saved. Align the two constants (single source).
35. **Ship opinionated default bindings** — [VALUE: low · EFFORT: S] — Nearly every `open_*` combo defaults to the empty string (`keyboard-shortcuts.js:11-13`) — the mechanism exists but ships fully unbound. A power product ships defaults (and #33's new actions should launch bound, e.g. to a `g`-prefix chord family) rather than requiring every user to discover the editor first.
36. **A global "?" shortcuts cheat-sheet** — [VALUE: med · EFFORT: S] — Only the image editor has a shortcuts overlay (`workspace/static/js/editor/keyboard-shortcuts.js:71`); the global, customizable map has zero discoverability. A `?` overlay rendered *from* the live keybind map (so custom binds show correctly) is the standard affordance.
37. **j/k + Enter list triage in Portal and digest** — [VALUE: med · EFFORT: M] — No applicant modal has any list-level keyboard handling (verified: Escape-to-close only; chat composer Enter). quick-wins #20 covers approve-keys *inside* the redline; the missing layer is row navigation — j/k (or arrows) to move a focus ring through Portal/digest rows, Enter to open, `a`/`d`/`s` for approve/decline/snooze on the focused row. Together with #25/#26 this is the "triage 20 roles in 60 seconds" experience.
38. **Refactor Ctrl+K into a provider registry (prerequisite for applicant entities)** — [VALUE: high · EFFORT: M] — ux-flows #8 asks for the palette outcome; the implementation reality is that the existing overlay is hard-wired end-to-end to conversation search — one fetch to `/api/search`, one renderer, one navigate (`workspace/static/js/search-chat.js:155-174`, `:111`). There is no seam to add a source. Refactor handleInput/renderResults/navigate into registered providers (conversations, surfaces, applicant entities), then the applicant provider is ~50 lines over #16's search endpoint. Ranked here because the *refactor* is the real cost the headline hides.
39. **Palette/shortcut *actions*, not just navigation — starting with global pause** — [VALUE: med · EFFORT: S] — Owner-scoped global pause/resume endpoints already exist (`workspace/routes/applicant_control_routes.py:92` `pause-all`/`resume-all`) — the master audit's Top-25 #7 ("global pause on the strip") is therefore cheaper than scoped, and the same endpoints belong in the palette (#38) and on a keybind as verbs ("Pause all campaigns", "Resume", "Approve all pending"). Verify current UI reachability of pause-all and wire it into every power entrance at once.

## Tier 5 — URL deep-links (the enumeration beyond ux-flows #1)

40. **A `link` field in the engine notification schema (the engine-side half of deep-linking)** — [VALUE: high · EFFORT: M] — ux-flows #1/#2 specify the front-door hash scheme (`#portal`, `#documents/<appId>`, `#remote/<appId>`) and that fan-out should embed URLs — but the engine's notification rows (`src/applicant/app/routers/notifications.py`) carry no target/link field at all, so there is nothing for the Portal toast, the Discord/email fan-out, *or* a webhook (#12) to embed. Add `link` (surface + entity id) to the notification payload at creation time in the engine services; the front-door then maps it to hashes. Without this, hash routing only helps manual navigation.
41. **The high-value hash targets the ux audit didn't enumerate** — [VALUE: med · EFFORT: S] — Beyond `#portal`/`#documents/<id>`/`#remote/<id>` (ux-flows #1) and `#debug/<tab>` (ux-flows #29), the power set needs: `#results` (the sharable/bookmarkable funnel — pairs with export #17), `#campaign/<id>/settings` (link from digest rows to "tune this campaign"), `#vault` (Settings-buried today, `index.html:1683-1688`), and `#activity`. Cheap once the router exists; listed so they land in the same PR.
42. **Register applicant modals in the minimize/dock registry** — [VALUE: med · EFFORT: S] — `modalManager`'s dock is keyed to a fixed `_LABELS` map (`workspace/static/js/modalManager.js:107-130`) that omits every applicant surface (ux-flows #37 flags parity). The power-user angle: dock registration is also what lets a half-done redline *survive* opening Remote — minimize instead of teardown — which no amount of hash routing gives you. Small map addition, big workflow continuity.

## Tier 6 — Automation rules & scheduled behavior (guardrailed — never auto-submit)

43. **Wire the dormant `Campaign.schedule` field (accepted, stored, never read)** — [VALUE: high · EFFORT: M] — `PUT /api/agent-runs/{cid}/config` accepts and persists a `schedule` dict on the campaign (`src/applicant/core/entities/campaign.py:43`) that **no code consumes** — `application/services/scheduler.py` never reads it. This is a silently-accepted no-op config (bug-adjacent: an API caller setting working hours believes it worked). Honor it in the tick gate and you get per-campaign run windows ("weekdays 9–5 only") — the recipe the brief calls quiet-hours-per-campaign — with the field already in the schema.
44. **Per-campaign quiet hours for notifications** — [VALUE: med · EFFORT: S] — Notification quiet hours exist but only globally (`adapters/notification/apprise_notifier.py:99-110,396-427`; configured via `setup.py:262,268`). Once #43 gives campaigns a schedule shape, let a campaign override the global window (e.g. the "urgent/US-timezone" campaign may ping at 22:00; the exploratory one never does).
45. **Guardrailed auto-approve rules for digest rows** — [VALUE: high · EFFORT: L] — No rules engine exists anywhere (verified: no auto-approve in engine or workspace — correctly, for submission). But *digest approval* is upstream of the stop boundary: a user-defined rule — "fit ≥ 0.85 AND source ∈ {Greenhouse} AND salary ≥ floor → auto-approve for pre-fill, cap 5/day" — only moves roles into the pre-fill queue; review-before-submit and the final gate remain core-enforced and untouched (the guard must derive ground truth server-side, never from the rule payload). Ship with dry-run preview ("this rule would have approved 7 of last week's 23") and a per-rule audit trail. This is the single biggest daily-time-saver a power user can be given safely.
46. **One-click deterministic block rules ("never this company again")** — [VALUE: med · EFFORT: M] — Decline feedback feeds *learned* taste (FR-DIG-5), but there is no user-visible hard blocklist distinct from learning — a power user who says "never Acme, ever" has no guarantee it sticks. Add an explicit exclusion list (companies, title keywords) applied deterministically at digest assembly, editable in the criteria surface, offered as a one-click action on decline.
47. **A weekly cadence value (plumbing for scheduled reports)** — [VALUE: med · EFFORT: S] — Digest/report cadence supports only `off`/`daily` (`application/services/scheduler.py:52-119`; `app/config.py:302-313`). product-gaps #33 and the master audit's weekly-recap item both presuppose a weekly tick that doesn't exist — add `weekly` to the cadence enum + scheduler gate first, or every weekly feature reimplements its own timer. (The workspace-side alternative is a scheduler recipe, #13 — do at least one.)
48. **Per-event-type notification routing** — [VALUE: med · EFFORT: M] — Channel fan-out is all-or-nothing per channel (`setup.py` channels config; apprise adapter): a user cannot say "digests → email, action-required → ntfy push, errors → Discord". A small event-type × channel matrix in Settings (default: current behavior) turns the notification ladder into an actual routing table — the power complement of quiet hours.

## Tier 7 — Multi-campaign muscle

49. **Clone a campaign** — [VALUE: med · EFFORT: S] — Campaign create takes a bare name (`src/applicant/app/routers/campaigns.py:57-95`); the settings surface offers rename/run-mode/throughput/exploration/archive + source toggles only (`workspace/static/js/applicantCampaignSettings.js:71-159`). "Same search, different city/title" — the most common second-campaign shape — means reconfiguring everything by hand. `POST /campaigns/{id}/clone` copying criteria, sources, throughput, and variant links (not history) + a Duplicate button.
50. **Campaign templates/presets** — [VALUE: low · EFFORT: M] — With #49 and profile export (#18), a step further: named presets ("Senior PM — remote US", "Staff Eng — EU") bundling criteria + sources + throughput, instantiable at create time. Serves career-pivoters running parallel personas (adjacent to product-gaps #22 but config-level, not resume-level).
51. **Owner-scoped per-campaign pause in Campaign Settings** — [VALUE: med · EFFORT: S] — Per-campaign pause/resume exists but only admin-gated in the ops surface (`workspace/routes/applicant_ops_routes.py` POST `/runs/{cid}/pause`·`/resume`, behind `_require_admin`; Debug UI), while Campaign Settings — the owner's surface — offers only archive (`applicantCampaignSettings.js:71-159`). The single-user self-hoster who isn't admin cannot pause one campaign without pausing all (`applicant_control_routes.py:92`). Re-home pause/resume (and run-now) as owner-scoped campaign controls.
52. **Compare campaigns (yield side-by-side)** — [VALUE: med · EFFORT: M] — Results is a single-campaign read-only view with a Refresh button (`applicantResults.js`; `applicant_results_routes.py` GET `""`), and Compare diffs applications/postings, not campaigns (`applicantCompare.js:106-110`). A user running "PM" vs "Eng" campaigns cannot see which converts — apps/day, approval rate, response rate per campaign in one table. The per-campaign stats already exist in the runs/insights data; this is a presentation join.
53. **Revisit the ops admin gate for the owner** — [VALUE: med · EFFORT: S] — Everything operational — run config, throughput, run-now, discovery source toggles, exploration budget — is `_require_admin` (`applicant_ops_routes.py`), a workspace-role concept that doesn't map to "the owner of this job search" on a family/multi-user box. Decide per-endpoint: throughput/run-now/sources are owner concerns; update-trigger stays admin. (Same product decision as the master audit's momentum-out-of-Debug theme, applied to *controls*.)

## The through-line

The buyer is a technical self-hoster, and the vendored front-door already contains everything such a
buyer expects — scoped tokens, a tool-using agent with an audited loopback, an MCP client, webhooks,
a cron/event scheduler — while the engine already exposes pause/resume, bulk resolve, snooze, run
controls, and even an MCP tool surface. Almost none of these are connected to each other, attributed
to an owner, fenced at the tool layer, or documented. The highest-leverage sequence: fix token owner
attribution (#3), add named agent tools + the `app_api` fence (#1–2), preconnect the engine's MCP
server (#7), bridge engine events to webhooks (#12), and add the search endpoint (#16) — five
medium-sized changes that turn ~twenty of the remaining items into small wiring tasks, and turn
"autonomous job-application engine" into a system a power user can script, schedule, and drive from
anywhere — with the submit boundary exactly as hard as it is today.
