# Notifications & Channels — end-to-end audit (exhaustive pass 2, lens 10)

> Scope: the full per-event pipeline — emit sites → `NotificationService` → `AppriseNotifier`
> ladder (in-app / Discord / email / ntfy) → engine `/api/notifications` → workspace portal proxy →
> Portal center / toasts / badge — plus quiet hours, dedup, digest email, Settings affordances.
> Dedup honored: morning-ritual timing, reward-carrying copy, badge-differentiates-wins,
> milestone emissions, toast deep-link click-through, batch-me-daily mode, contextual desktop-permission
> prompt (engagement #5/#8/#19/#20/#21/#22/#48) are NOT repeated here.

## The taxonomy, as built (reference for the findings)

**Emitted kinds (grep of every emit site):** material/variant review decisions
(`material_service.py:1400,1505`), final-approval decisions (`final_approval_service.py:55`),
pre-fill blocked pings — account step / 2FA / detection / emergency handoff
(`prefill_service.py:1956,2086`), digest-ready ping (`notification_service.py:109`), digest email
(`digest_service.py:286`), daily status update (`status_update.py:71`), essentials nudge
(`essentials_nudge.py:73`), scheduler-stall error (`scheduler.py:338`), stuck-application error
(`agent_loop.py:718`), channels test (`app/routers/setup.py:310`), follow-up sends
(`post_submission_service.py:175` — dead, see #1), cautious-mode detection ping
(`prefill_service.py:944` — dead, see #2).

**Rendered kinds (Portal):** exactly four — `action` (skipped; represented by pending rows),
`digest`/`info` ("Update"), `error` ("Heads up") — from `_classify`
(`apprise_notifier.py:461-476`) and `_NOTIF_KIND_LABEL` (`applicantPortal.js:832-836`). No
positive/success kind exists anywhere in the enum, classifier, or renderer.

## Tier 1 — broken pipes (emitted-but-never-delivered, delivered-but-wrong)

1. **Follow-up notifications are dead code twice over** — [VALUE: high · EFFORT: S] — `send_scheduled_follow_ups` calls `self._notification.notify(title=..., body=..., deep_link=...)` but `self._notification` is the `NotificationService` (`post_submission_service.py:29`, wired at `container.py:868`), which has **no `notify` method at all** — every call raises `AttributeError`, is swallowed by `except Exception: continue`, the follow-up is never appended to `sent` and never transitions to `FOLLOWING_UP`, so due follow-ups retry forever. And nothing ever calls `send_scheduled_follow_ups` (no scheduler hook, no router — grep hits only the definition). Anchor: `src/applicant/application/services/post_submission_service.py:169-191`.

2. **Cautious-mode detection pause pings a method that doesn't exist** — [VALUE: high · EFFORT: S] — the planner-path `blocked_detection` stop calls `self._notification.notify_pending(pending_action)` inside `try/except: pass`, but no class in the tree defines `notify_pending` (grep: zero definitions). The one moment the spec calls out as needing the human (FR-PREFILL-6 cautious pause) emits nothing on any channel; the user finds out only if they open the Portal. Anchor: `src/applicant/application/services/prefill_service.py:942-946`.

3. **CRITICAL urgency is renderable-but-never-emitted — CAPTCHA/takeover waits are held by quiet hours** — [VALUE: high · EFFORT: S] — the port defines `CRITICAL` precisely for "live-takeover / captcha hand-off ... NEVER deferred by quiet hours" (`ports/driven/notification.py:21-26`) and the adapter fully implements it (`apprise_notifier.py:344-354,448`), but **no emit site ever uses it**: blocked/2FA/emergency/detection pings go out as default-NORMAL `Notification`s (`prefill_service.py:1956,2086`), so an agent blocked at 2am waits silently until the quiet window ends while the Settings copy promises "anything urgent always comes through right away" (`applicantOnboarding.js:511`). Map the handoff ping kinds to CRITICAL. Anchor: `src/applicant/ports/driven/notification.py:26` + `prefill_service.py:1956`.

4. **The digest email is dispatched as raw HTML with no `body_format` — most SMTP recipients see markup soup** — [VALUE: high · EFFORT: S] — `send_email` wraps the rendered `<h1>/<table>` string into `Notification(body=html)` and `_send_real_dispatch` calls `client.notify(title, body)` with no `body_format=html` and no format hint on the Apprise URL, so Apprise's default text handling delivers literal `<table border='1'>` source (or strips it, per plugin). There is also no plain-text alternative part. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:543-549` + `notification_service.py:179-185`.

5. **Every deep link in every outbound channel is a relative path — unclickable and unroutable** — [VALUE: high · EFFORT: M] — `_send_real_dispatch` appends `deep_link` verbatim to the Discord/email/ntfy body (`apprise_notifier.py:544-545`), and every emit site passes engine-relative paths: `/digest?campaign=…` (`notification_service.py:144`), `/activity` (`status_update.py:78`), `/wizard` (`essentials_nudge.py:81`), `/applications/{id}` (`post_submission_service.py:178`), `review_base_url?document_id=…` (`material_service.py:1381`). No public base URL exists in config (`app/config.py:249-255` has channels only), and the front-door SPA has no hash routes for any of these paths anyway — so a Discord push reads "Tap to open the redline review\n/review?document_id=x" with nothing to tap. Fix needs (a) a `PUBLIC_URL` setting, (b) a per-message engine-path → front-door-surface mapping. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:544-545`.

6. **The Portal renders `deep_link` on nothing — info rows are dead ends even in-app** — [VALUE: high · EFFORT: S] — the router ships `deep_link` in every row (`app/routers/notifications.py:51`) but `_renderNotifRow` renders title/body/Dismiss only — no link, no click handler, no affordance (`applicantPortal.js:838-853`). A "Daily digest ready" row and its "tap to see what was searched" body sit above a Dismiss button and nothing else. (Engagement #48 covers toast click-through; this is the persistent row itself.) Anchor: `workspace/static/js/applicantPortal.js:838-853`.

7. **Engine restart silently drops every queued/held escalation rung** — [VALUE: high · EFFORT: L] — the entire ladder state (`_sent` deliveries, unfired Discord-hold/email/quiet-held rungs), the in-app inbox, the dismissed set, the digest-ready per-day marker, and the digest-email dedup set are all process memory (`apprise_notifier.py:225-249`, `notification_service.py:37`). Restart during the 30s hold or the 15-min email window → the push never goes out (the DB-backed pending action survives, the notification doesn't); restart mid-quiet-hours → everything held is lost, and "Deliver now" flushes nothing. The spec's durable-orchestration story (DBOS `recv` for final approval) never covers the notifier. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:223-249`.

8. **Restart also re-emits: same-day duplicate digest email + ready ping** — [VALUE: med · EFFORT: M] — the once-per-(campaign, UTC-day) guards live only in memory (`notification_service.py:37,153` and `_sent_emails` at `apprise_notifier.py:236`), so a mid-day restart re-runs `deliver()` and sends a second "Your daily digest" email and a second ready ping the same day. The comments call this "defense in depth" but the depth is one process lifetime. Persist the day-markers (they already have DB-shaped keys). Anchor: `src/applicant/application/services/notification_service.py:125-158`.

9. **`dedup_key` does not deduplicate emission — and call sites believe it does** — [VALUE: high · EFFORT: M] — `notify()` with an existing dedup key *replaces* the `_sent` entry and immediately fires fresh rungs (`apprise_notifier.py:562-580`); the key only enables `expire()` cross-channel cancel (FR-NOTIF-3). But the scheduler-stall alert comments "even if this fired twice the notifier collapses it to a single operator alert" (`scheduler.py:344-346`) and both nudge services call their per-day keys "a no-op at the notifier" (`notification_service.py:200-202,227-229`) — all false: a second `notify()` with the same key re-pushes every channel. Either implement emit-side dedup for an active key or fix the three comments and audit every re-emit path. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:562-580`.

10. **"Deliver now" over-delivers: it force-fires future escalation rungs, not just quiet-held ones** — [VALUE: med · EFFORT: S] — `deliver_now` fires *every not-yet-fired rung regardless of its scheduled `due_at`* (`apprise_notifier.py:672-687`, `_fire_due(force=True)` at 712-722). A user tapping "Deliver now" to release an overnight digest also instantly triggers the 15-minute email backstop for every open decision (email they'd never otherwise get if they acted in-app) and skips the Discord presence pre-empt (line 727-730). Filter the force-flush to rungs whose `due_at` has passed but were quiet-deferred. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:672-687`.

11. **Web presence only counts if the digest surface is open — Portal activity doesn't pre-empt Discord** — [VALUE: med · EFFORT: S] — the FR-NOTIF-2 presence heartbeat (`POST /api/applicant/email/presence`) is started solely from the digest module (`applicantDigest.js:893-937`); the Portal — the actual notification center where the user answers decisions — never signals presence, nor does any other surface. A user actively working the Pending queue is "absent" to the engine, so the Discord push fires anyway 30s later. Boot the heartbeat globally (it's already tab-focus/activity aware). Anchor: `workspace/static/js/emailLibrary/applicantDigest.js:860-937`.

12. **Snoozing a Portal item does not quiet its notification ladder** — [VALUE: med · EFFORT: M] — the pending-actions router claims "snoozing an item also defers its re-notification until the wake time" (`pending_actions.py:130-131`), but snooze only reschedules the DB row; the notifier's already-queued rungs (Discord hold, 15-min email) for that decision's dedup key still fire — the user snoozes "remind me tomorrow" and gets the email 15 minutes later. Snooze should `expire()` the ladder and re-arm on wake. Anchor: `src/applicant/app/routers/pending_actions.py:130-131` vs `apprise_notifier.py:651-670`.

13. **No submission/response/interview success notification exists — the positive half of the taxonomy is missing at the *pipe* level** — [VALUE: high · EFFORT: M] — `submission_service.py` has zero notify calls; the Portal's own header comment promises "digest ready / **submitted** / errors" (`applicantPortal.js:59,104-106`) but nothing ever emits a submitted notification, and `_classify` has no branch that could label one (`apprise_notifier.py:461-476`). Beyond the listed milestone/celebration items: add a `success` kind end-to-end (enum → classifier → `_NOTIF_KIND_LABEL` → styling) so good news is even *representable* in the channel taxonomy. Anchor: `src/applicant/adapters/notification/apprise_notifier.py:461-476`.

14. **The Settings test notification is classified as an error and renders as a red "Heads up" row** — [VALUE: med · EFFORT: S] — `/channels/test` sends `IMMEDIATE` urgency (`setup.py:310-317`), `_classify` maps IMMEDIATE → `error` (`apprise_notifier.py:472-473`), so the in-app copy of every test lands in the Portal with the danger accent and "Heads up" tag — the user's very first notification experience looks like a failure. Add a `test`/`info` classification (key off `dedup_key="channels-test"`) or send NORMAL and exempt it from quiet hours explicitly. Anchor: `src/applicant/app/routers/setup.py:310-317` + `applicantPortal.js:839`.

15. **ntfy never carries a priority header — urgent action alerts and FYIs arrive identical on the phone** — [VALUE: med · EFFORT: S] — the adapter adds bare `ntfy://` URLs to Apprise (`apprise_notifier.py:539-542`) with no `priority=` param and no urgency→priority mapping, so an IMMEDIATE "agent is stuck" push and a test ping render with the same default priority — no bypass-DND, no distinct sound, defeating the point of a phone-push channel. Map NORMAL→default, IMMEDIATE/CRITICAL→`urgent` (Apprise supports `?priority=` on ntfy URLs). Anchor: `src/applicant/adapters/notification/apprise_notifier.py:539-542`.
