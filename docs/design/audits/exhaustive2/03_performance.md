# Applicant — Performance & Perceived Speed (exhaustive2 · lens 03)

> **Lens.** Real measured code paths across both apps: module loading, static-serving economics,
> polling costs, the glass/mesh GPU bill, proxy-hop latency, engine query patterns behind the hot
> polled endpoints, boot time, memory growth, and the perceived-speed tricks not yet used.
>
> **Baseline numbers (measured on this tree).** `workspace/static/style.css` = **1.24 MB**
> (35,937 lines) · `index.html` = **232 KB** (2,626 lines) · `static/js/` = **5.9 MB** across ~162
> ES modules · **47 scripts referenced directly by index.html (~2.5 MB raw)**, only **5**
> `modulepreload` hints · **116** `backdrop-filter` · **337** `box-shadow` · **140** `@keyframes`
> (68 `infinite`) · **60** `transition: all` · **6** `will-change` · **0** `contain:` /
> `content-visibility` in the whole stylesheet.
>
> **Dedup.** Excludes items already landed or listed elsewhere: hidden-tab poll pausing
> (`applicantCore.js:142` `pollVisible` — DONE), the `_fetchJSON` AbortController/15s timeout
> (`applicantCore.js:31-48` — DONE), spinner/loading-kit reuse (`applicantCore.js:95` — DONE), and
> the prior audit's coalesce-strip-and-badge / debounce / low-power-mode / CDP-measurement items.
> Format: `N. **Title** — [VALUE · EFFORT] — rationale + anchor.`

## Ranked findings

1. **No response compression anywhere (no GZip/Brotli middleware)** — [VALUE: high · EFFORT: S] — `workspace/app.py` registers CORS, SecurityHeaders, RequestTimeout, and Auth middleware (app.py:100/119/161/406) but never `GZipMiddleware`. The 1.24 MB CSS, 232 KB HTML, 5.9 MB of JS, and every polled proxy JSON payload go over the wire raw. One line — `app.add_middleware(GZipMiddleware, minimum_size=1000)` — cuts text transfer ~5–8× for every user forever. The engine FastAPI app should get the same.

2. **~47 eagerly loaded module scripts on every boot, including 400 KB+ modules for rare surfaces** — [VALUE: high · EFFORT: M] — `index.html:2572-2624` loads 44 `type="module"` tags (47 unique script files, ~2.5 MB raw before the import graph pulls the rest of the 5.9 MB). The heaviest are eager: `document.js` (423 KB), `slashCommands.js` (255 KB, via imports), `notes.js` (233 KB), `emailLibrary.js` (233 KB), `settings.js` (225 KB), `chat.js` (219 KB), `documentLibrary.js` (218 KB), plus rarely opened surfaces (`applicantDebug.js`, `applicantCompare.js`, `applicantGallery.js`, `cookbook.js`, `compare/index.js`, `admin.js` at index.html:2590-2620). Convert rare-surface modules to lazy `import()` behind their launcher clicks (each already boots via a launcher-wiring loop, so the seam exists) and keep only the shell + Portal path eager. This is the single biggest cold-boot lever.

3. **Per-request `httpx.AsyncClient` construction on every workspace→engine proxy call** — [VALUE: high · EFFORT: M] — `ApplicantEngineClient.__init__` builds a fresh `httpx.AsyncClient` (new connection pool → new TCP connect, zero keep-alive reuse) per call (`workspace/src/applicant_engine.py:148`), and ~180 call sites do `async with ApplicantEngineClient()` per request (e.g. `applicant_portal_routes.py:247`, `applicant_activity_routes.py:112/142/175/217`, all of `applicant_admin_routes.py`). Hold one app-lifetime client in `app.state` and inject it; every proxy hop then rides pooled keep-alive connections.

4. **Portal `/pending` is a serial 2M+1 engine fan-out per poll** — [VALUE: high · EFFORT: M] — the home-base feed calls `list_campaigns()`, then a sequential `await engine.list_pending_actions(cid)` per campaign in a for-loop (`workspace/routes/applicant_portal_routes.py:249-268`), then `_onboarding_gap_item` loops another sequential `await engine.onboarding_state(cid)` per campaign (`:161-169`). M campaigns ⇒ up to 2M+1 serial HTTP hops on the highest-traffic route — and the 60 s badge poll re-pays it forever. `asyncio.gather` the fans now; add a single engine "all-campaigns pending" aggregate endpoint as the real fix.

5. **The 60 s badge poll downloads the full pending payload (and triggers #4) just to compute a count** — [VALUE: high · EFFORT: S] — `refreshBadge` (`workspace/static/js/applicantPortal.js:1275`, polled at `BADGE_POLL_MS = 60000`, `:45`, wired `:1508`) fetches `${API}/pending` — full shaped rows plus the notifications list — to set an integer badge. Add a `/pending/count` (or `HEAD`-style summary) endpoint returning `{count}` cheaply from the indexed `(campaign_id, resolved)` column; reserve the full fetch for an open Portal.

6. **Engine digest GET re-scores every posting in the campaign, unbounded, per request** — [VALUE: high · EFFORT: M] — `GET /api/digest/{campaign_id}` → `build_digest` loops `postings.list_for_campaign` with no LIMIT (`src/applicant/adapters/storage/repositories.py:388-394`) and calls `score_fn(posting, criteria)` per row (`application/services/digest_service.py:106-136`); the docstring itself anticipates "1000+ viable roles". The Portal loads this on every open (`applicantPortal.js:1466`). Cache the built digest payload per campaign/day and use a viability-filtered, column-projected query.

7. **Every gated engine request does an uncached `app_config` SELECT on the shared boot Session** — [VALUE: high · EFFORT: M] — `require_llm_configured` (`src/applicant/app/deps.py:296-305`) is a dependency on nearly all routers and drills to `config_store.get(_LADDER_KEY)` → a real `SELECT` (`adapters/storage/app_config_store.py:58-69`) against the single container-level boot Session (`:39-53`) on every call — including every 45–60 s poll. The ladder changes rarely: cache in-process, invalidate on `configure_llm`/`set_tiers`.

8. **`require_automated_work` scans all campaigns + per-campaign readiness on every polled request** — [VALUE: high · EFFORT: M] — `is_automated_work_allowed()` (`application/services/setup_service.py:745-757`) invokes the `_onboarding_gate` closure that iterates `storage.campaigns.list()` and computes `is_ready_to_apply` (criteria load + résumé check) per campaign (`app/container.py:481-485`, `onboarding_service.py:333-373`) — on `agent_status`, digest, agent-runs: all polled surfaces. Cache with a short TTL / invalidate on onboarding writes.

9. **Per-bubble `backdrop-filter` on every chat message** — [VALUE: high · EFFORT: M] — `body.theme-frosted .msg-ai` / `.msg-user` each carry `backdrop-filter: blur(22px) saturate(180%) !important` (`workspace/static/style.css:34602`, `:34614`). A long transcript = dozens of independent offscreen backdrop composites per scrolled frame — the most-repeated glass surface in the app. Make bubble frost a flat translucent fill and let only the window chrome sample the backdrop.

10. **Animated mesh: `blur(54px)` re-painted every frame via `background-position` animation** — [VALUE: high · EFFORT: M] — `css/meshGradient.css:57` puts `filter: blur(54px) saturate(1.55) brightness(1.14)` on a larger-than-viewport layer (`inset:-25%`, `background-size:150%`) and `:71/:76-79` animates it infinitely on **background-position** — a full paint + 54 px gaussian per frame, continuously, on every surface. The `::after` ray adds a second animated `blur(30px)` layer (`:67/:74`). Animate `transform` on a pre-blurred layer instead so the blur rasterizes once and frames are composite-only.

11. **`index.html` (232 KB) re-read + re-templated from disk per navigation, no conditional GET** — [VALUE: high · EFFORT: M] — `serve_index` → `serve_html_contained` does a full `open().read()` + `html.replace("{{CSP_NONCE}}", nonce)` per request with no ETag/304 (`workspace/app.py:902-928`, `src/app_helpers.py:88`); nine deep-link routes share it (`app.py:930-964`). The per-request nonce makes the whole 232 KB body uncacheable. Cache the file bytes on mtime, and move the nonce into a tiny inline bootstrap so the bulk shell can 304.

12. **Blanket `Cache-Control: no-cache` forces ~165 revalidation round-trips per page load** — [VALUE: high · EFFORT: M] — `_RevalidatingStatic` stamps `no-cache` on every `.js/.css/.html` (`workspace/app.py:424-428`). Starlette does answer 304 (bytes saved), but each of the ~162 modules + CSS still costs a conditional-GET RTT per load — and the service worker is **network-first** for JS/CSS (`static/sw.js:5-6`), so the SW cache never hides that latency either. A short `max-age=60` (or content-hashed URLs) keeps the no-restart dev property while collapsing the storm.

13. **Zero CSS containment anywhere — offscreen windows and long lists pay full style/layout/paint** — [VALUE: high · EFFORT: M] — 0 hits for `contain:`/`content-visibility` in all 35,937 lines of style.css. Background `.ow-window`s, the chat transcript, and the memory/skills/entity lists are uncontained. Add `content-visibility: auto` + `contain-intrinsic-size` to non-focused windows and long scroll lists; `contain: layout paint` on modal bodies. Pairs multiplicatively with #9.

14. **Stacked glass-on-glass `backdrop-filter` chains** — [VALUE: high · EFFORT: M] — `.ow-window` samples at `blur(30px)` (`style.css:32828/32867`), chrome cards/toasts/modals at `blur(22px)` (`:32827/:32848`), buttons add `blur(10px)` (`:33092/:33170`), bubbles another `blur(22px)` (#9). A toast-in-window or button-in-window stacks 2–3 nested backdrop composites, each re-sampling the layer beneath. Enforce "only the outermost surface in a nesting chain samples the backdrop".

15. **`/api/applicant/features` (boot-critical) = 3 serial sync HTTP calls on 3 fresh clients, uncached** — [VALUE: high · EFFORT: M] — `compute_features` runs healthz → `get_sync("/api/setup/status")` → `get_sync("/api/dormant-surfaces")` serially (`workspace/src/applicant_features.py:330-348`), each opening a short-lived `httpx.Client` (`applicant_engine.py:907/927`). It is threadpool-offloaded (`applicant_routes.py:44`) so the loop is safe, but the front-end reads it on every boot and it changes rarely. Drop the redundant healthz, share one client, and memoize ~10 s (stale-while-revalidate).

16. **Continuous full-viewport canvas rAF loops allocate gradients + call `getComputedStyle` per frame** — [VALUE: high · EFFORT: M] — theme.js runs unbounded `requestAnimationFrame(draw)` loops for 7+ wallpaper patterns (`theme.js:1662/1743/1826/1915/1966/2025/2093`); the embers loop creates a fresh `createRadialGradient` **per particle per frame** (`:2117`), and each loop re-reads `getComputedStyle(document.documentElement)` every frame (`:1664/:1899/:1963/:2011/:2079`). Cache colors across frames, reuse gradient objects, and pause the loop when the tab is hidden.

17. **Portal first paint is serialized behind the notifications fetch** — [VALUE: high · EFFORT: S] — `_load` awaits `_loadNotifs()` *before* `_renderList(body)` (`workspace/static/js/applicantPortal.js:1426-1429`), despite the comment claiming independence — a slow inbox delays the action rows the user opened the Portal for. Render the pending rows immediately, fold notifications in when they arrive.

18. **Workspace startup does RAG/vector/manager init at import time, before serving** — [VALUE: med · EFFORT: L] — module import runs `get_rag_manager()` (ChromaDB connect, `workspace/app.py:497`), `initialize_managers()` which can rebuild the memory vector index inline (`src/app_initializer.py:60-67`), and TTS/STT init (`app.py:533/674`), plus eager imports of the heavyweight agent-tooling modules. Warmups are already deferred to the startup event — move vector rebuild + RAG connect there too so first-accept is fast.

19. **Engine per-request factory rebuilds ~25 services per poll** — [VALUE: med · EFFORT: M] — any dependency through `get_request_services` (`src/applicant/app/deps.py:36-74`) triggers `_build_request_services`, constructing the full Learning/Scoring/Digest/Prefill/Material/Submission/… graph (`app/container.py:1256-1420`) for a single polled `GET`. Only Session-bound repos need per-request isolation — build the stateless services once and rebind, or construct lazily per accessed dep.

20. **Activity surface re-fetches the campaign list on every poll across 4 endpoints** — [VALUE: med · EFFORT: M] — `/status`, `/intent`, `/runs`, `/snapshot` each call `_owner_campaigns` → `list_campaigns` then one more engine call = 2 serial hops each (`workspace/routes/applicant_activity_routes.py:112-121/142-151/175-184/217-226`); the always-visible strip polls two of them at 45 s (`applicantActivity.js:34/477`). Memoize owner→campaign resolution briefly or collapse onto one consolidated engine status read.

21. **No backoff-on-idle in any poll loop** — [VALUE: med · EFFORT: S] — badge (60 s, `applicantPortal.js:45`) and strip (45 s, `applicantActivity.js:34`) tick at fixed cadence forever, even when payloads have been identical for hours on an idle self-hosted box. `pollVisible` (`applicantCore.js:142`) is the perfect chokepoint to add exponential idle backoff (e.g. ×2 up to 5 min after N unchanged responses) with instant reset on visibility/interaction/mutation.

22. **`pollVisible` adopted by only 4 modules — other pollers still run unguarded intervals** — [VALUE: med · EFFORT: S] — only `applicantPortal/Activity/Results` use the visibility-aware helper; `assistant.js:454` (status poll), `appkitDecision.js:830`, `calendar.js:3064` (30 s today-tick), and `censor.js:109` roll bare `setInterval`s that keep firing in hidden tabs. Migrate them to `pollVisible`.

23. **Engine gallery payload embeds every material's full document body** — [VALUE: med · EFFORT: M] — `AdminQueryService.gallery` inlines `"content": d.content` (full résumé/cover-letter text) for every material plus every screenshot row, unpaginated (`src/applicant/application/services/admin_query_service.py:103-127`; unbounded `documents.list_for_campaign` at `repositories.py:538-545`). The grid renders only metadata (`applicantGallery.js:141-218`). Return refs/excerpts; fetch bodies on demand; add limit/offset.

24. **No thumbnails or dedicated cacheable image route for pre-fill screenshots** — [VALUE: med · EFFORT: M] — screenshots surface only as `page_ref` Text strings inside gallery/history JSON (`admin_query_service.py:103-111`, model `adapters/storage/models.py:227-234`) with no separate `GET /screenshots/{id}` carrying content-type + `Cache-Control`/ETag, and no thumbnail variant. If `page_ref` ever carries data-URLs, polled list responses become multi-MB. (The workspace's own generated-image route shows the right pattern: content-hash filename + `immutable`, `workspace/app.py:476-480`.)

25. **30 s read-timeout applied to hot interactive polls** — [VALUE: med · EFFORT: S] — `_DEFAULT_TIMEOUT = httpx.Timeout(connect=3.0, read=30.0, …)` (`workspace/src/applicant_engine.py:46`) covers features/portal/activity reads; a stalled engine pins each poll request ~30 s (holding a fresh connection each, per #3). Split a short interactive budget (~5–8 s) from the generous LLM-write budget (the documents route already models this with its own turn timeout, `applicant_documents_routes.py:243`).

26. **Full-list `innerHTML` rebuild on every Portal refresh — no diffing, state wiped** — [VALUE: med · EFFORT: M] — each refresh re-renders the entire action+notification list as one string (`applicantPortal.js:879`), discarding scroll position/focus and re-parsing identical DOM on every 60 s tick or resolve. Key rows by id and patch only changed rows (or at minimum skip render when the payload hash is unchanged — cheap stale-check).

27. **Engine pending-actions list: no LIMIT, Python-side sort per poll** — [VALUE: med · EFFORT: S] — `list_open` selects all unresolved rows (indexed but unbounded, `repositories.py:783-790`) and `list_with_metadata` sorts in Python each call (`pending_actions_service.py:175-200`). Add pagination/LIMIT and push priority ordering into SQL.

28. **Engine `agent_status` poll sums the day's runs by decoding JSON rows in Python** — [VALUE: med · EFFORT: S] — `count_pipelines_started_on` loads every run row for the UTC day and sums `stats["pipelines_started"]` in Python (`repositories.py:932-951`), plus `campaigns.get` + `agent_runs.latest` = 3 queries per 45 s poll. Promote the counter to a real column and `SUM()` it, or maintain a daily counter.

29. **Engine setup `/status` recomputes readiness + channels + tiers + suggested attributes per poll** — [VALUE: med · EFFORT: S] — `get_status` → `_status_dict` re-runs config reads, `apply_readiness()` campaign scans, and the suggested-attributes reporter (which invokes advanced-learning suggestion logic) on every call (`src/applicant/app/routers/setup.py:125-134`, `container.py:759-771`). Polled, rarely changing — memoize with short TTL, invalidate on config writes.

30. **Digest build hydrates full posting `description` Text blobs it never emits** — [VALUE: med · EFFORT: M] — `list_for_campaign` loads whole `JobPostingModel` rows including `description` (`models.py:159`, `repositories.py:388-394`) though `build_digest` emits only title/company/url/score (`digest_service.py:109-118`). Thousands of rows × multi-KB descriptions of wasted IO per build. Use a column-projected query on the digest path.

31. **Engine DB pool left at defaults (5 + 10 overflow) vs a 40-thread request pool** — [VALUE: med · EFFORT: S] — `make_engine` sets `pool_pre_ping` but no `pool_size`/`max_overflow`/`pool_recycle` (`src/applicant/adapters/storage/session.py:24-30`), while per-request sessions (Starlette threadpool ≈ 40 workers), the scheduler tick session, the boot Session, and audit-log emissions all draw from it. Under load, checkouts stall behind 15 connections. Size the pool to the threadpool.

32. **Digest `deliver` materializes N pending actions with a dedup SELECT + commit each** — [VALUE: med · EFFORT: S] — `deliver` loops rows calling `digest_approval` per row (`digest_service.py:257-269`); each `materialize` runs `find_open_by_dedup` then `commit()` (`pending_actions_service.py:57-73`) — N SELECTs + N commits in one request. Batch-insert in a single transaction.

33. **Scheduler rebuilds the full ~25-service graph every tick** — [VALUE: med · EFFORT: M] — `_build_tick_services` reconstructs the whole Learning/Scoring/Discovery/Digest/Prefill/Material/Submission/AgentLoop wiring against a fresh session each interval (`src/applicant/app/container.py:1123-1250`, invoked `scheduler.py:182`). Per-tick Session isolation is by design (CLAUDE.md), but the session-*independent* construction can be cached and only the storage-bound repos rebound per tick.

34. **`appkitGlass` MutationObserver re-scans the whole document during chat streaming** — [VALUE: med · EFFORT: M] — `watchMounts` observes all body subtree additions (`workspace/static/js/appkitGlass.js:929`); each batch schedules `collectTargets` → `document.querySelectorAll` over 8 selectors (`:844`) with a `getComputedStyle(el)` visibility check per candidate (`:853`). Token streaming = hundreds of insertions = repeated full-document scans + forced style reads. Narrow the observer to mount roots and use `offsetParent` for visibility.

35. **`getBoundingClientRect` on every `pointermove` over glass surfaces** — [VALUE: med · EFFORT: S] — the specular-rim tracker reads `el.getBoundingClientRect()` per pointermove (`appkitGlass.js:1051`); writes are rAF-batched (`:1057-1058`) but the read can force layout while streaming dirties styles. Cache the rect on pointerenter/scroll/resize.

36. **Sticky headers carrying `backdrop-filter` re-composite the scroll every frame** — [VALUE: med · EFFORT: S] — the chat top bar carries the glass backdrop (`style.css:32535/:34542`) and ~15 `position: sticky` sites sit over scrolling content (e.g. `:17625`, `:20029`, `:30039`). A sticky element with a backdrop-filter must re-sample the moving content beneath it per scrolled frame. Swap to an opaque/solid fill while scrolling.

37. **`transition: all` ×60 on dense interactive rows/buttons** — [VALUE: med · EFFORT: M] — 60 occurrences, mostly `0.15s` on list rows and buttons (`style.css:6753/:6799/:6928/:6950/:6997/:23676/:25463/:25490`). `all` animates any property change — including layout-affecting ones — and blocks the compositor fast-path. Enumerate the real properties (`background, color, opacity, transform`).

38. **Transitions/hints on layout-triggering properties** — [VALUE: med · EFFORT: M] — transitions animate geometry in hot spots: `width/opacity/padding` (`style.css:409`), `left/top/width/height` (`:859`), `width…padding…border-radius` (`:2333/:2408`), `max-width/margin-left` (`:2438`), `top` (`:1799/:9759`); and `will-change: padding` (`:11687`) wastes a layer on an uncompositable property. Prefer `transform`+`opacity`; move `will-change` to the transform-animated layers.

39. **Infinite `box-shadow` pulse/breathe keyframes repaint per frame** — [VALUE: med · EFFORT: S] — several always-on animations animate `box-shadow` (paint-bound): rail-notes pulse (`style.css:753-754`), notification pulse (`:15242-15246`), status pulse (`:32200`), plus unread-breathe/reminder-glow variants. Replace with an `opacity`-animated pseudo-element ring (composite-only).

40. **`background-attachment: fixed` full-bleed layers defeat scroll compositing** — [VALUE: med · EFFORT: S] — 3 sites: the tiled dots patterns (`style.css:227/:233`, 20–24 px tiles repainted full-viewport on scroll) and a wallpaper-region rule (`:35069`). Use a `position: fixed; z-index:-1` layer instead.

41. **4-layer stacked box-shadows on every glass window re-rasterize during drag** — [VALUE: med · EFFORT: S] — the shared chrome rule stacks a rim + two inset rims + the two-shadow float var on every `.ow-window/.modal-content/.admin-card/.toast` (`style.css:32857-32861`, `--ow-glass-float` `:32819`). Promote dragged windows to their own layer or collapse to one float shadow during drag.

42. **Per-window exact-fit SVG displacement filters — cap concurrent refracted surfaces** — [VALUE: med · EFFORT: M] — each eligible surface gets its own feImage→feDisplacementMap(→blur→colorMatrix→specular feBlend) chain via inline `backdrop-filter: url(#…)` (`appkitGlass.js:612-764`, applied `:800`). Maps are correctly cached per size bucket and only re-built on a debounced ResizeObserver (`:482/:906` — good), but every unique large window is still its own offscreen refraction pass. Lower `activeMaxSurfaces` so background windows fall back to plain frost.

43. **Inter fonts not preloaded → FOUT + first-paint layout shift** — [VALUE: med · EFFORT: S] — the three `@font-face` declarations use `font-display: swap` (`index.html:195-197`) but there is no `<link rel="preload" as="font">` (only 5 JS `modulepreload`s at `:229-233`), so fonts are discovered after CSS parse and the page visibly reflows from fallback to Inter. Preload the 400/600 woff2s.

44. **Workspace boot-critical HTML depends on third-party CDN (KaTeX/Mermaid)** — [VALUE: low · EFFORT: S] — `index.html:204-205` pulls KaTeX + Mermaid from jsdelivr (async, integrity-pinned — good), but on a self-hosted LAN box these stall or 404 offline and contend with first-paint network. Self-host under `/static/lib/` (highlight.js already is, `:192`) and lazy-load only when a math/diagram block renders.

45. **Timeout middleware wraps every request — including static 304s — in `asyncio.wait_for`** — [VALUE: low · EFFORT: M] — `_RequestTimeoutMiddleware` creates a task + 45 s timer per request including `/static` (`workspace/app.py:147-161`), stacked with 3 more pure-Python `BaseHTTPMiddleware` layers (`:100/:119/:406`). With ~165 asset revalidations per load (#12) that's measurable pure overhead. Exempt `/static` and cheap GETs, or go ASGI-native.

46. **CSP nonce generated for every static/API request that never uses it** — [VALUE: low · EFFORT: S] — `SecurityHeadersMiddleware` runs `secrets.token_hex(16)` per request including `/static/*` (`workspace/core/middleware.py:107-113`). Skip nonce generation for static paths.

47. **Auth middleware does two locked session lookups per request** — [VALUE: low · EFFORT: S] — the cookie path calls `validate_token` then `get_username_for_token` (`workspace/app.py:396/:402`), each taking the sessions lock and re-running expiry checks (`core/auth.py:427-450/:452-475`). Merge into one `resolve(token) → username|None`. (Verified: no DB/bcrypt on the cookie hot path — already good.)

48. **Generated-image route runs a DB ownership query per image request** — [VALUE: low · EFFORT: S] — `serve_generated_image` opens a fresh `SessionLocal` and queries the gallery row per image (`workspace/app.py:450-460`); a grid of N thumbnails = N sessions + N queries on open (mitigated by the `immutable` cache header on repeat views, `:476-480`). Batch-authorize per gallery load or cache filename→owner briefly.

49. **No pagination passthrough on proxied history/log reads** — [VALUE: med · EFFORT: S] — the engine client hardcodes `limit=200` for application history and `limit=100` for logs with no client-driven paging (`workspace/src/applicant_engine.py:576/:593`), and the portal feed concatenates all campaigns' items unbounded (`applicant_portal_routes.py:278-290`). Add limit/offset passthrough so polled surfaces stop re-downloading full histories.

50. **LLM keepalive loop opens fresh clients to up to 5 endpoints every 60 s forever** — [VALUE: low · EFFORT: S] — `_keepalive_loop` → `_warmup_endpoints` builds a new `httpx.AsyncClient(timeout=5.0)` per endpoint per cycle (`workspace/app.py:1058-1086`). Reuse one client; make the cadence configurable.

51. **Update log poll re-fetches the whole status+log payload every 3 s** — [VALUE: low · EFFORT: S] — the update surface polls `${OPS}/update` at `POLL_MS = 3000` and re-renders the full response each tick (`workspace/static/js/applicantUpdate.js:28/:142-158`); as the log grows the payload grows quadratically over the run. 52. **Adaptive-ink pass interleaves computed-style reads with streaming writes** — [VALUE: low · EFFORT: M] — the veil/ink sampler reads `getComputedStyle` per adaptive element (`theme.js:1727/:1747/:1815/:1884`), debounced (`appkitGlass.js:1808` pattern) but still interleaving reads with token-streaming DOM writes → read-write-read thrash. Batch all reads before any writes per pass.
