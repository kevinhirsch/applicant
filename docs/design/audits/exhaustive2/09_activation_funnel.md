## A. Pre-login: what the front door promises

1. **The landing page never mentions job applications** — [VALUE: high · EFFORT: M] — `workspace/static/landing.html:338-346` sells "Your own AI workspace… chat, autonomous agents, tools, model serving, email, research". The word "job", "apply", "resume", or "application" appears nowhere in 739 lines. The user's first screen promises a different product than the OOBE delivers ("Welcome to Applicant… connect a model… start applying"). Rewrite the hero + features grid around the actual value proposition: an agent that finds, tailors and files job applications for you 24/7, with review-before-submit.
2. **No "Sign in" / "Open the app" CTA anywhere on the landing page** — [VALUE: high · EFFORT: S] — every CTA is `#start` (a git-clone codeblock) or GitHub (`landing.html:347-350,563-566`). A user standing in front of a *running* instance has no path from the marketing page to `/login`. Add a nav + hero "Sign in" button pointing at `/login`.
3. **Landing "Get started" ends at `git clone`** — [VALUE: med · EFFORT: S] — `landing.html:557-575` tells an already-deployed visitor to clone the repo. For the self-host case the real happy path is `bash scripts/proxmox-deploy.sh` / `scripts/install.sh` (per repo README); for a deployed instance it's "log in". The one command shown is the least useful of the three.
4. **Gag testimonials undermine the trust-critical ask** — [VALUE: med · EFFORT: S] — `landing.html:355-433` ("Loved by enterprises", a screaming cyclops, fake five-star reviews). Charming for a hobby workspace; corrosive for a product that will hold your legal name, EEO data, salary floor and submit things under your identity. Keep the humor, but move it below a serious "what it does / what it never does" section that reuses the wizard's `NEVER_DOES` list (`applicantOnboarding.js:100-105`).
5. **Landing promises features the front-door gates or hides** — [VALUE: med · EFFORT: S] — the features grid (`landing.html:444-490`) advertises Cookbook, Compare, Email Assistant, Deep Research — vendored-workspace features — and none of the engine's actual differentiators (digest, redline review, live takeover, stealth pre-fill). The pre-login story and the post-OOBE product don't intersect.
6. **Preview panels reference `chat.webm`/`email.mp4` that don't ship** — [VALUE: low · EFFORT: S] — `landing.html:522,531` point at video sources not present in `workspace/static/`; the panels silently fall back to placeholder text ("Hover to take a closer look" then nothing plays). Either ship real captures of the digest/redline/takeover loop (the most convincing pre-login artifact possible) or drop the section.
7. **Landing page hard-codes its own theme, ignoring the app's saved theme** — [VALUE: low · EFFORT: S] — `landing.html:10-26` re-declares tokens; `login.html` carefully boots the user's saved theme + bg pattern (`login.html:540-558`). Landing → login → shell shifts visual identity twice in the first minute.
8. **No screenshot/product-shot of the daily payoff pre-login** — [VALUE: med · EFFORT: M] — the `shotrow`/`previews` scaffolding exists (`landing.html:235-277`) but shows placeholders. The single highest-converting pre-login asset for this product is a digest morning-view screenshot: "8 matches found overnight, 3 ready for your approval."

## B. Login & first shell load

9. **First-run admin-create mode is undiscoverable from the landing page** — [VALUE: med · EFFORT: S] — `/login` auto-detects unconfigured auth and flips to "First-time setup — create your admin account" (`login.html:313-319,348-350`), which is good — but nothing pre-login tells the operator that visiting `/login` is the next step after `install.sh`. One line on the landing hero ("Fresh install? Create your admin account →") closes the loop.
10. **Password policy surfaces only on submit** — [VALUE: low · EFFORT: S] — the 8-char minimum is validated after the user submits (`login.html:405-410`); no inline hint under the password field in setup/signup mode. Show requirements up front; losing a first-time admin on a password error is pure friction.
11. **No "what happens next" between account-create and the wizard** — [VALUE: low · EFFORT: S] — after setup-mode auto-login the page prefetches sessions/features/settings and hard-redirects to `/` (`login.html:435-451`); the user lands in a strange shell for ~1-2s before the wizard overlay mounts (`app.js:1402-1405` awaits a status fetch first). A one-frame "Setting things up…" splash owned by the wizard would remove the flash of an unexplained, locked workspace.
12. **Engine-down at boot silently skips the wizard AND the Portal** — [VALUE: high · EFFORT: M] — `maybeLaunchOnboarding()` catches the status fetch failure and returns `false` (`applicantOnboarding.js:1630-1632`); `app.js:1404-1409` then opens the Portal only if the wizard didn't show — but on engine-unreachable the user gets the bare vendored workspace with every Applicant nav item lock-toasted "Finish setup to unlock this" (`app.js:1360`) and *no way to know the engine is down, nor any retry*. First-boot race (engine still migrating while UI is up) reads as a broken product. Detect the unreachable case distinctly and show a "The application engine is starting… retrying" banner with backoff.
13. **The lock-toast lies when setup is actually complete** — [VALUE: med · EFFORT: S] — the click-guard toast is always "Finish setup to unlock this" (`app.js:1360`), and the title fallback "unlocks once the Applicant engine is configured" (`app.js:1323`) — both wrong for `present_but_disabled` (dormant build capability) and for engine-down-after-setup. Thread the section's real state into the message.
14. **First shell load drops a brand-new user into the generic chat shell behind the wizard** — [VALUE: low · EFFORT: S] — the blocking overlay covers a fully-rendered workspace (chat, editor, email icons all visible but locked behind the modal). Peripheral UI competes with the wizard; a dimmed/blurred backdrop (the `ow-window` glass exists, `applicantOnboarding.js:151-158`) tuned to actually obscure the shell would focus the first 5 minutes.

## C. The OOBE wizard: structure & navigation

15. **"Skip for now →" is the same button as "Finish" — skip semantics are invisible** — [VALUE: med · EFFORT: S] — the persistent nav renders one button that morphs (`applicantOnboarding.js:275-284`). A user on "Connect a model" clicking "Skip for now" lands on Profile; clicking again "Finish"es into an "Almost there" scold (`_finish`, `applicantOnboarding.js:1576-1600`). Nothing while skipping says *what* skipping costs ("skipping this means Applicant can't start"). Annotate the skip button per-step with the consequence.
16. **Step rail claims "2 of 3" but shows no time estimate** — [VALUE: med · EFFORT: S] — the rail (`_renderRail`, `applicantOnboarding.js:179-192`) shows ✓/number + title only. The profile step hides an 12-section interview (`INTAKE_SECTIONS`, line 91-96); the honest label is "Your profile (~10 min, or 2 min with a resume)". Progress honesty caps abandonment: users quit unbounded forms, not long ones.
17. **The intake's 12 sub-steps are invisible from the rail** — [VALUE: med · EFFORT: S] — the rail shows 3 top-level steps; inside "Your profile" a second counter appears ("Profile step 3 of 12", `applicantOnboarding.js:1238`). Two disjoint progress systems, and the outer rail never advances during the longest stretch of the funnel. Nest a mini-progress bar (or "step 3/12" chips) into the rail's profile tab.
18. **Rail steps aren't clickable — Back is the only way to revisit** — [VALUE: low · EFFORT: S] — `_renderRail` renders inert spans (comment at `applicantOnboarding.js:182-184`). A user 9 sections deep who wants to fix the model must click Back ~10 times. Completed steps should be clickable jump targets (the `_finish` jump buttons prove the mechanic exists, line 1593-1600).
19. **Esc / initModalA11y can dismiss the "blocking" wizard mid-step** — [VALUE: med · EFFORT: S] — the overlay wires `initModalA11y(_overlay, _dismiss)` (`applicantOnboarding.js:1640-1642`), passing `_dismiss` as the close callback — so whatever close gestures the shared a11y helper binds (Esc) tear the wizard down without confirmation, mid-upload included. Deliberate re-entry works (resumable), but an accidental Esc mid-resume-parse silently discards in-progress form edits (unsaved section data lives only in the DOM). Confirm-before-dismiss, or save the section draft first.
20. **Mid-section form data is lost on any re-render** — [VALUE: med · EFFORT: M] — intake sections save only on "Save & continue" (`applicantOnboarding.js:1252-1268`); Back, Esc-dismiss, tab-close, or an engine hiccup during save discards everything typed in the section (work-history highlights are the painful case). Persist a per-section draft to `localStorage` keyed by campaign+section and rehydrate on render.
21. **Closed-tab re-entry resumes the wizard but not the sub-section scroll state** — [VALUE: low · EFFORT: S] — resume logic reopens at the first incomplete *section* (`applicantOnboarding.js:1203-1205`) — good — but a user who saved 6 of 7 work-history entries and quit returns to a blank repeat-form (see #34). State honesty: show what was already saved on re-entry ("You've added 6 roles").
22. **`_ensureCampaign` silently names the campaign "My job search" with no rename moment** — [VALUE: low · EFFORT: S] — `applicantOnboarding.js:123-134`. The user never chooses or sees the campaign name during OOBE, then meets "My job search" in every later surface. One inline editable field on the criteria section fixes the disconnect.
23. **Wizard completion state can wedge on a legacy flag mismatch** — [VALUE: med · EFFORT: S] — `maybeLaunchOnboarding` gates on `llm_configured && onboarding_complete` (`applicantOnboarding.js:1634`), but the STEPS treat profile as done at `apply_ready || onboarding_complete` (line 80). A user who skips the profile forever (allowed, by design) gets the wizard re-launched on *every* login even though the rail shows all steps green — the "done" wizard reopens at its last step endlessly. Align the launch predicate with the step predicates (`llm_configured && (apply_ready || onboarding_complete || user_dismissed)`), or persist an explicit "user finished the wizard" bit.
24. **No engine-side dismissed/completed timestamp → no funnel telemetry at all** — [VALUE: med · EFFORT: M] — nothing records when the wizard was first shown, per-step enter/leave, or completion time, so time-to-first-value is unmeasurable and every activation experiment is blind. Add lightweight step-timing events to the existing setup-status advance calls (`POST /setup/advance/<step>` already exists, `applicantOnboarding.js:246`) — the engine can timestamp them for a real funnel report.

## D. Step 1 — Connect a model

25. **The endpoint manager is mounted raw with zero wizard-specific guidance** — [VALUE: high · EFFORT: M] — `_renderLLM` drops Settings' full endpoint manager into the step (`applicantOnboarding.js:347-364`): a power-user surface (add local / add remote / per-model toggles / enable switches) with no "recommended path" framing. A first-timer needs a fork: "I have Ollama on this machine" (one click, see §7 headline #2) / "I have an API key" (one field) / "Show me the full manager". Progressive disclosure over the lifted admin panel.
26. **Cloud key paste has no validate-before-save feedback loop in the wizard** — [VALUE: high · EFFORT: M] — the wizard's Save & continue reads whatever the manager has (`getSelectedLlmEndpoint`, `applicantOnboarding.js:373`) and only then posts to the engine gate; a bad/expired API key surfaces as the generic "Could not connect this model to the application engine: …" (line 404-408). The manager's own "Test" exists in Settings but nothing in the wizard forces or even suggests a test-before-continue. Wire a mandatory quick inference ping ("Say hi" round-trip with visible latency) before the gate save, with provider-specific failure hints (401 → "check the key", 404 model → "pick a model the endpoint lists").
27. **Error message doesn't distinguish the four distinct failure modes** — [VALUE: med · EFFORT: S] — `applicantOnboarding.js:402-410` collapses (a) no endpoint added, (b) endpoint disabled, (c) endpoint has no models, (d) engine gate rejected, into two strings. Each has a different fix; say which one happened ("Your endpoint is added but disabled — flip the switch above").
28. **API key is never sent to the engine — silent dependency on endpoint-manager state** — [VALUE: med · EFFORT: M] — the gate save posts `api_key: ''` (`applicantOnboarding.js:396-400`) on the assumption the key "already lives with the saved endpoint". If the engine's provider client needs its own key (engine ≠ workspace credential store), the user discovers it only later, deep in the first discovery run. Verify at the gate: have the engine do a live token-count/completion before reporting `llm_configured`.
29. **`_inferProvider` folds every unknown cloud provider into "openai"** — [VALUE: med · EFFORT: S] — `applicantOnboarding.js:426-439`. Anthropic-compatible proxies, Groq, Mistral, OpenRouter, Azure-OpenAI (different auth header + api-version) all become `openai`; a wrong guess produces a downstream engine failure the user can't trace back to this dropdown-less moment. Show the inferred provider in the UI before saving ("Connecting as: OpenAI-compatible") with an override select.
30. **No model-quality guidance at the single most consequential choice** — [VALUE: high · EFFORT: M] — the wizard treats a 3B local quant and a frontier cloud model identically, but the whole product's output quality (match scoring, tailoring, cover letters) hangs on this pick. `applicantModelLadder.js` exists precisely to rank/climb models — surface its verdict here: "This model is below the recommended tier for resume tailoring — fine to start, expect rougher drafts" (see §E).
31. **No cost expectation for cloud keys** — [VALUE: med · EFFORT: S] — a user pasting an OpenAI/Anthropic key gets no estimate of what 24/7 discovery + tailoring costs ("typically $X–Y/day at 5 applications/day"). Cost shock is a day-2 churn driver unique to autonomous agents; one static sentence at the paste moment is cheap insurance.
32. **The manager relocation can strand Settings if teardown misfires** — [VALUE: low · EFFORT: S] — `mountEndpointManager` physically moves the Settings node into the wizard and `_restoreEndpointManager` moves it back (`applicantOnboarding.js:333-345,412`); the restore is "best-effort" in a `try{}catch{}`. A thrown restore leaves Settings' Services panel with a hole. Harden with a `finally` on every exit path (dismiss already does, `applicantOnboarding.js:1603-1609`; `_renderStep`'s non-llm branch covers most).

33. **Ladder/verdict is invisible during the step where it matters** — [VALUE: med · EFFORT: M] — see #30; anchor: `workspace/static/js/applicantModelLadder.js` renders tiering elsewhere, `_renderLLM` never imports it.

## F. Step 2 — Your profile (intake, per field)

34. **`repeat: true` sections render only ONE entry's fields — multi-entry work history is silently impossible** — [VALUE: high · EFFORT: M] — `work_history`, `education`, `references` are declared repeatable (`applicantOnboarding.js:1098-1131`) but `_renderIntakeSection` renders `spec.fields` once with no "Add another role" control, and `_collectForm` flattens inputs into a single object (`applicantOnboarding.js:1190-1197`). A career of N jobs becomes one job unless the resume parse filled the rest server-side. This kneecaps every downstream Workday pre-fill. Implement the repeat UI (add/remove entry cards) or at minimum render the parsed entries read-only with per-entry edit.
35. **Prefilled-vs-empty fields are visually identical** — [VALUE: med · EFFORT: S] — after resume parse the section forms render `saved[f.name]` (`applicantOnboarding.js:1236`) with no "from your resume" affordance. The user can't tell what the machine guessed (needs review) from what they typed (trusted). A subtle "parsed" chip per prefilled field turns proofreading into a scan.
36. **Identity email isn't prefilled from the logged-in account** — [VALUE: med · EFFORT: S] — the workspace knows the user's email; the identity section (`applicantOnboarding.js:1052-1063`) starts blank. Prefill `email` (and `preferred_name` from the username) before the resume even lands — zero-cost derived fields.
37. **`address` is one free-text line feeding structured ATS forms** — [VALUE: med · EFFORT: M] — Workday wants street/city/state/zip/country as separate fields; collecting "Mailing address" as one string (`applicantOnboarding.js:1059`) guarantees a lossy split later in pre-fill. Structure it now (or parse+confirm the split inline).
38. **Phone has no country-code guidance, `type: 'text'`** — [VALUE: low · EFFORT: S] — `applicantOnboarding.js:1058`; `type="tel"` + placeholder "+1 555 …" prevents the classic unfillable-Workday-phone-widget failure.
39. **LinkedIn/portfolio URLs aren't validated or normalized** — [VALUE: low · EFFORT: S] — `applicantOnboarding.js:1060-1061`; "linkedin.com/in/me" vs full https URL matters to form-fillers. Normalize on save.
40. **Work-authorization country is free text** — [VALUE: med · EFFORT: S] — `authorized_country` (`applicantOnboarding.js:1067`) will be matched against job postings and ATS dropdowns; "USA"/"United States"/"US" divergence creates silent filter misses. A country select (datalist) with ISO names is a 20-line fix.
41. **The sponsorship question ignores the authorized=no case** — [VALUE: low · EFFORT: S] — if `authorized: no` the flow still asks visa status flat (`applicantOnboarding.js:1064-1072`); conditional reveal ("Not authorized → which visas could work?") mirrors how real ATS forms branch and prevents contradictory answers the fabrication guard must later reconcile.
42. **Work mode is free text with the options in the label** — [VALUE: med · EFFORT: S] — "Preferred work mode (remote / hybrid / onsite)" as `type: text` (`applicantOnboarding.js:1078`) invites "wfh preferably", unusable for discovery filters. Three checkboxes (multi-select — most people accept 2 of 3) feed the matcher clean enums.
43. **Salary floor: free text, no currency coupling, no privacy note** — [VALUE: med · EFFORT: S] — `salary_floor`/`desired_salary`/`currency` are three uncoupled text fields (`applicantOnboarding.js:1090-1097`). "$120k", "120,000", "120000 USD" all land differently in the comparator that gates discovery. Numeric input + currency select + one line on how the floor is used ("jobs below this are filtered out, never disclosed to employers unless a form requires it").
44. **Target titles as comma-separated text vs the tag-input the workspace already has** — [VALUE: low · EFFORT: S] — `titles`/`adjacent_titles` (`applicantOnboarding.js:1085-1088`); the memory/criteria surfaces already render tag chips — lift that input for consistency and clean tokenization.
45. **References collected during OOBE — prime deferral candidate** — [VALUE: med · EFFORT: S] — references (`applicantOnboarding.js:1122-1131`) are needed by a minority of applications, at submit time, not before first value. Move the section out of the default path ("Applicant will ask when an application needs one"), shortening the interview by 1/12.
46. **EEO section asks for four sensitive categories at minute ~8 of a product the user hasn't seen work yet** — [VALUE: med · EFFORT: S] — `applicantOnboarding.js:1140-1149`; defaults are honest ("decline to self-identify") — good — but the ask itself is deferrable to the first application that actually presents an EEO page, where the context is self-evident. Defer-and-ask-in-flow beats asking on faith (§7 #8 covered the inline explanation; this is the sequencing beneath it).
47. **The EEO "prefer to answer" select + free-text detail is a worse control than the real EEO enums** — [VALUE: med · EFFORT: S] — `_fieldHTML` type `eeo` renders a two-option select and an optional text field (`applicantOnboarding.js:1178-1186`); real forms have fixed option sets (e.g. the standard race/ethnicity categories, veteran status phrasings). Free text guarantees a mismatch the agent must fuzzy-map later — under a *never guess* policy (`NEVER_DOES`, line 103-104). Offer the canonical option lists.
48. **The EEO detail input never rehydrates its saved value** — [VALUE: med · EFFORT: S] — the detail input is rendered with `value=""` always (`applicantOnboarding.js:1184`), so revisiting the section shows the choice but not the answer — a returning user can't verify what they wrote without re-entering it. Bind `saved[f.name + '__detail']`.
49. **`campaign_criteria` free-text is the last section — the coldest field gets the tiredest user** — [VALUE: med · EFFORT: S] — "Describe your ideal next role" (`applicantOnboarding.js:1150-1157`) is the highest-leverage input for discovery quality and arrives at section 12/12. Move it right after `target_roles`, and prefill a draft from the parsed resume + titles ("Here's what I inferred — edit").
50. **No per-section "why we ask" microcopy** — [VALUE: low · EFFORT: S] — only `eeo` and `work_history` have a `desc` (`applicantOnboarding.js:1100,1142`). One line each ("References: some applications require 2-3; we'll only use them with your approval") converts interrogation into collaboration.
51. **yes/no selects default to "—" with no required-ness signal** — [VALUE: low · EFFORT: S] — `_fieldHTML` yesno (`applicantOnboarding.js:1170-1176`); sections save with everything blank (no client validation at all), and the user discovers gaps only via the engine's `apply_missing` list later. Mark the truly-required-to-apply fields (per the engine's own ground truth) with the standard required marker.
52. **Section save failures dead-end with a generic message and keep the user's data hostage in-DOM** — [VALUE: med · EFFORT: S] — `applicantOnboarding.js:1262-1267`: "Could not save." + re-enabled button; no retry-with-backoff, no draft persistence (see #20), no distinction between validation 4xx and engine-down 5xx.

## G. Resume upload & the LaTeX accept/reject gate

53. **Resume "Continue" stays disabled until preview succeeds — TeX absence blocks the whole step** — [VALUE: high · EFFORT: S] — after upload, the flow runs font-detect then `_buildPreview` and only then enables Continue (`applicantOnboarding.js:1420-1423`); on preview failure the button is never re-enabled — only the generic "Preview unavailable: …" renders (`applicantOnboarding.js:1515-1517`) and the user's sole exit is the global "Skip for now" (which skips the *whole profile step context*, not just the preview). Since the engine silently degrades when `xelatex`/LibreOffice are missing (CLAUDE.md "Runtime dependencies"), a missing image dep turns into a wedged OOBE. Enable Continue regardless; the preview is an enhancement, not a gate.
54. **The accept/reject choice has no visual preview — it's a text description of fidelity** — [VALUE: high · EFFORT: M] — `_buildPreview` shows "(N page(s)). Looks like a faithful match." + notes (`applicantOnboarding.js:1495-1504`) and asks the user to choose "Use this version" vs "Keep my original" *without showing either document*. This is the one decision that changes what gets sent under the user's name. Render the PDF inline (an `<iframe>`/`<object>` of the converted artifact) side-by-side or at least a first-page thumbnail.
55. **Accept/reject buttons give no state feedback beyond a status span, and don't advance** — [VALUE: med · EFFORT: S] — clicking either sets a small text ("Using the polished version.", `applicantOnboarding.js:1505-1514`) but Continue was already enabled before the choice; nothing marks the decision as pending/required, and a user can Continue without choosing at all — what's the default then? Make the choice explicit (two cards, one gets selected) and reflect the persisted state on re-entry.
56. **Inline font-install gate requires ALL missing fonts before Continue enables** — [VALUE: med · EFFORT: S] — `_renderInlineFontPrompt` enables its Continue only when `installedCount >= missing.length` (`applicantOnboarding.js:1307-1314`), and the counter increments only on success — with no per-font skip ("use a fallback font"). A user without the licensed `.ttf` (most people don't have their resume's font as a file!) is walled. Offer "Continue with substitutes" and explain the visual cost.
57. **The font-file ask is expert-level with no help** — [VALUE: med · EFFORT: S] — "Upload font file for **Calibri**" (`applicantOnboarding.js:1286-1293`) assumes the user knows what a .ttf is, where fonts live on their OS, and the licensing implications. Add a "where do I find this?" tip and auto-suggest metric-compatible free fallbacks (Carlito for Calibri etc. — the TeX image likely ships them).
58. **Double upload of the same resume: once to `/base-resume`, again to `/fonts/detect`** — [VALUE: low · EFFORT: S] — `readResume` re-posts the file it just uploaded (`applicantOnboarding.js:1408-1412`); on slow links this doubles the wait of the funnel's heaviest moment. Let the engine run font detection on the already-stored document.
59. **The parse-success line reports a raw count with no reviewability** — [VALUE: low · EFFORT: S] — "Read 47 details from your resume" (`applicantOnboarding.js:1398`); 47 of what? Link it forward: "identity ✓, 4 roles, 2 degrees, 23 skills — review them in the next steps."
60. **Conflict resolution defaults to keeping the interview value with no "apply all from resume"** — [VALUE: low · EFFORT: S] — `_renderConflicts` (`applicantOnboarding.js:1451-1478`) checks "Keep" by default and posts one confirm per conflict sequentially in a loop; ten conflicts = ten radio pairs. Bulk actions ("Use resume for all") + one batched POST.
61. **Upload accepts `.md`/`.txt` resumes but the preview pipeline is LaTeX/docx-shaped** — [VALUE: low · EFFORT: S] — accept list at `applicantOnboarding.js:1352`; the conversion preview's fidelity language ("keeps its look") is meaningless for plaintext. Branch the copy, skip the font-detect for text formats.

## H. Wizard completion, first-run defaults, and the dead-air gap

62. **Completion screen ends at a full stop — "Get started" just closes the overlay** — [VALUE: high · EFFORT: M] — `_finish` → `_dismiss` → `refreshApplicantFeatures()` (`applicantOnboarding.js:1567-1569,1603-1620`); no first discovery is triggered, no Portal/Activity is opened (app.js's Portal-open only ran at boot, before the wizard), and the user is deposited on the generic chat shell with newly-unlocked nav icons and zero direction. This is the exact Beat-2 dead-air the master audit names (§7 #3/#10) — the per-step gap here: `_dismiss` should hand off to `openApplicantPortal()` + kick a discovery run + show "I'm searching now — first digest by HH:MM."
63. **No first-digest ETA anywhere** — [VALUE: high · EFFORT: S] — the single most anxiety-reducing sentence a delegation product can utter ("Your first digest arrives tomorrow at 8:00") appears nowhere in the wizard, completion screen, or Portal empty state. The digest schedule is engine config; surface it at completion.
64. **The "Almost there" screen only lists the LLM as missing — profile gaps ride the readiness banner** — [VALUE: low · EFFORT: S] — `_finish` builds `missing` from `llm_configured` alone (`applicantOnboarding.js:1558-1560`) while `apply_missing` items are prose in `readyLine` (line 1562-1566) with no jump-back buttons to the specific profile section that would clear each item. Deep-link each missing essential to its intake section.
65. **First campaign is created with engine defaults the user never sees** — [VALUE: med · EFFORT: S] — `_ensureCampaign` POSTs `{name}` only (`applicantOnboarding.js:131`); daily-cap/continuity/aggressiveness defaults are invisible until the user finds Campaign Settings. Beneath §7 #9: show the effective defaults on the completion screen as a receipt ("I'll look for up to N roles/day; change this in Campaigns").
66. **Nothing schedules a "first 24 hours" check-in** — [VALUE: med · EFFORT: M] — the engine has notification fan-out + quiet hours from minute one; a canned T+24h notification ("Here's what I did in day one: searched X, matched Y — approve your first 3") is the standard activation-retention play and needs only a one-shot scheduled notification at wizard completion.
