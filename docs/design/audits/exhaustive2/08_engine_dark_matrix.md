# Exhaustive Audit 2 — Lens 08: Dark Engine Capability (the systematic BE→FE matrix)

**Lens.** The prior audit's #1 finding was "built but dark." This pass makes that sweep systematic and
exhaustive: every engine router endpoint traced through workspace proxy → JS consumer → nav/feature
registry; every application-service public method with no router; every entity field computed but never
rendered; every config knob with no Settings UI; every adapter wired in `container.py` but unreachable;
every scheduler/tick behavior invisible to the user.

**Method.** Grounded greps of `src/applicant/app/routers/*.py` (31 routers, ~140 endpoints),
`workspace/src/applicant_engine.py` (client), the 23 `workspace/routes/applicant_*_routes.py` proxy
files, all `workspace/static/js/applicant*.js` consumers, `workspace/src/applicant_features.py`
(section registry), `src/applicant/application/services/*.py` (50 services),
`src/applicant/core/entities/*.py` (27 entities), `src/applicant/app/config.py` (~75 knobs),
`src/applicant/app/container.py`, and `scheduler.py`/`agent_loop.py`.

**Dedup.** Per the brief, these Wave 1–2 items are NOT repeated: learning summary
funnel/per-source/signature (Results surface), per-campaign pause/resume (global pause shipped),
submission-snapshot pre-submit preview, activity stats/recap/momentum. The post-submission
tracker/follow-ups/ghosting headline is taken a level deeper here (exact methods, fields, and wiring
chains) rather than restated.

## B. Ranked findings

### B1 — Broken or dead wiring found during the trace (fix first; minutes each)

1. **`applicantReachability.js` is a facade over three 404s** — [VALUE: high · EFFORT: S] — the module is loaded on every page (`workspace/static/index.html:2624`) and exports `ensureSubmittable` → `POST /api/applicant/documents/ensure-submittable` (real proxy is `/api/applicant/documents/applications/{application_id}/ensure-submittable`, `applicant_documents_routes.py:334`), `fetchLearnedCriteria` → `GET /api/applicant/criteria/{cid}/learned` (no `/api/applicant/criteria` router exists — live path is under `/api/applicant/memory`), and `deliverDigestNow` → `POST /api/applicant/digest/deliver-now` (no such route; live paths are `applicant_email_routes.py:291` and `applicant_portal_routes.py:393`). All three URLs 404, and no JS module imports any of the three exports — the "reachability wiring" file is itself unreachable AND wrong. Anchor: workspace/static/js/applicantReachability.js:9-21.
2. **`ensure-submittable` (auto-heal missing materials) has no working caller** — [VALUE: high · EFFORT: S] — engine `POST /api/documents/applications/{id}/ensure-submittable` (src/applicant/app/routers/documents.py:419) rebuilds whatever material an application is missing; proxy exists (`applicant_documents_routes.py:334`); the only JS reference is the broken helper in item 1 → a one-click "fix this application's documents" action the review surface never offers. Wire a real button in `documentLibrary.js` review cards + Portal blocked-material rows. Anchor: src/applicant/app/routers/documents.py:419.
3. **Chat-lane application actions are dead proxies** — [VALUE: low · EFFORT: S] — `applicant_chat_routes.py:395/403/411` proxy request-final-approval / resume-account-step / resume-detection-step for the chat surface, but `applicantChat.js` only calls `campaigns/confirm/confirm-criteria/message` (+ pending-actions); the remote lane owns these actions. Either wire chat action chips to them or delete the routes. Anchor: workspace/routes/applicant_chat_routes.py:395.
4. **`memory/attributes/acquire-missing` proxy is dead code** — [VALUE: low · EFFORT: S] — `applicant_memory_routes.py:302` proxies the engine's acquire-missing (`attributes.py:161`), but zero JS calls it — the Portal's own `/missing-attribute` route (`applicant_portal_routes.py:431`) is the live lane. Consolidate or delete. Anchor: workspace/routes/applicant_memory_routes.py:302.
5. **Duplicate digest-deliver lane, one side unused** — [VALUE: low · EFFORT: S] — `applicant_email_routes.py:210` (`POST /digest/{cid}/deliver`) has no JS caller; `applicant_email_routes.py:291` (`/campaigns/{cid}/digest/deliver`, used by `emailLibrary/applicantDigest.js:982`) is the live one. Remove the dup before someone "fixes" the wrong lane. Anchor: workspace/routes/applicant_email_routes.py:210.

### B2 — The post-submission lifecycle, a level deeper (no engine router exists for ANY of this)

The prior audit said "the tracker has no surface." Here is exactly what is built and exactly what wiring is missing. `PostSubmissionService` is fully implemented and tested, and **not one of its eleven public methods is importable from any router** (grep: 0 router imports).

6. **Tracker read-model: `poll_status()` per application** — [VALUE: high · EFFORT: M] — `PostSubmissionService.poll_status(application_id)` (src/applicant/application/services/post_submission_service.py:51) returns the live tracker state (`submitted → awaiting-response → following-up → rejected/ghosted → archived`); `GET /api/outcomes` is a bare index stub (`{"surface": "outcomes"}`, src/applicant/app/routers/outcomes.py:38). Wiring: replace the stub with a campaign-scoped list (`GET /api/outcomes/{campaign_id}` → per-application `{status, last_event, days_since_submission}`), add owner proxy `workspace/routes/applicant_tracker_routes.py`, render an applied→interview→offer board (nav next to Results). Anchor: src/applicant/application/services/post_submission_service.py:51.
7. **Follow-up scheduling + the send queue** — [VALUE: high · EFFORT: M] — `schedule_follow_up(application_id, template, delay_hours, subject, body)` (post_submission_service.py:149) and `send_scheduled_follow_ups(now)` (post_submission_service.py:169) implement a full timed follow-up queue over the `FollowUp` entity (`status SCHEDULED/SENT`, `scheduled_at`, `sent_at`, `subject`, `body` — src/applicant/core/entities/follow_up.py:23-31) — no router, no proxy, no JS; the user can never see "follow-up to Acme drafts Friday," approve it, or cancel it. Wiring: `GET/POST /api/follow-ups/{campaign_id}` + `POST /api/follow-ups/{id}/cancel` → owner proxy → a Follow-ups tab on the tracker board + Portal rows for due drafts. Anchor: src/applicant/application/services/post_submission_service.py:149.
8. **Follow-up drafting (`FollowUpService.draft_followup`)** — [VALUE: high · EFFORT: M] — `draft_followup(...)` (src/applicant/application/services/followup_service.py:48) plus `followup_is_due(...)`:36 and the `due_after_days` policy:31 generate the message body itself — the emotional "it nudged Acme for me" beat. Dark end-to-end; the review-before-send loop should reuse the existing document review/turn UI (`documentLibrary.js`) per lift-and-shift. Anchor: src/applicant/application/services/followup_service.py:48.
9. **Ghosting detection (`check_ghosting` + `SilenceService`)** — [VALUE: high · EFFORT: M] — `check_ghosting(campaign_id, sla_days=14)` (post_submission_service.py:115) walks submissions and emits `GhostingSignal(sla_days, submission_age_days, detail)` (src/applicant/core/entities/ghosting_signal.py:13-14) via `SilenceService.is_likely_ghosted` (silence_service.py:53) — no router, and the 14-day SLA is a hardcoded call-site default with no Settings knob. Wiring: scheduler tick calls it (see §B7 item 60) → notification + tracker "ghosted" column; SLA slider in Settings. Anchor: src/applicant/application/services/post_submission_service.py:115.
10. **Rejection email scanning never meets the inbox the workspace already has** — [VALUE: high · EFFORT: M] — `scan_email_for_rejection(email_subject, email_body, application_id)` (post_submission_service.py:92) + `RejectionService.classify_message`/`classify_status_page` (src/applicant/application/services/rejection_service.py:66,78) are built, and the front-door already polls mail (`workspace/routes/email_pollers.py`, `emailInbox.js`) — the two are never connected. Wiring: `POST /api/outcomes/applications/{id}/scan-email` (or an internal-token lane the workspace poller calls) → outcomes recorded → tracker/Portal badge. Anchor: src/applicant/application/services/post_submission_service.py:92.
11. **Manual "they rejected me" capture** — [VALUE: high · EFFORT: S] — `process_rejection_signal(application_id, source, signal_text, confidence, detail)` (post_submission_service.py:76) transitions state + persists a `RejectionSignal` (`signal_text`, `confidence` — src/applicant/core/entities/rejection_signal.py:20-21) — no router; the learning loop's strongest negative signal depends on the user being able to report it in one tap. Wiring: `POST /api/outcomes/applications/{id}/rejection` → tracker row button. Anchor: src/applicant/application/services/post_submission_service.py:76.
12. **`enter_post_submission` / `advance_to_awaiting_response` transitions never fire from any surface** — [VALUE: med · EFFORT: S] — post_submission_service.py:31 and :46 are the entry points of the state machine; nothing user-reachable invokes them, so even a submitted application never *enters* the tracker. Wire them into the existing final-submit confirmations (`remote.py:292/323` handlers). Anchor: src/applicant/application/services/post_submission_service.py:31.
13. **`archive()` — no way to close out an application** — [VALUE: med · EFFORT: S] — post_submission_service.py:213 archives an application; no router/proxy/JS → dead rows accumulate forever in every future tracker view. Wiring: `POST /api/outcomes/applications/{id}/archive` → swipe/button on tracker rows. Anchor: src/applicant/application/services/post_submission_service.py:213.
14. **One-tap outcome capture is admin-only** — [VALUE: high · EFFORT: S] — `POST /api/outcomes/applications/{id}/mark-submitted` (src/applicant/app/routers/outcomes.py:74) and `/detect` (:43) exist on the engine but are proxied ONLY behind `_require_admin` (`applicant_admin_routes.py:333`, `:352`) and consumed only by the Debug modal (`applicantDebug.js`). The owner — the person who knows they got the interview — has no owner-scoped lane. Wiring: add to the tracker proxy, buttons on tracker/Portal rows. Anchor: workspace/routes/applicant_admin_routes.py:333.
15. **Per-application outcome timeline is Debug-only** — [VALUE: med · EFFORT: S] — `GET /api/outcomes/applications/{id}/log` (src/applicant/app/routers/outcomes.py:93) is proxied admin-only (`applicant_admin_routes.py:272`) and rendered as a comma list in `applicantDebug.js:357`; `OutcomeEvent.type/source` (src/applicant/core/entities/outcome_event.py:46-47) deserve a real owner-visible timeline on each tracker row. Anchor: src/applicant/app/routers/outcomes.py:93.

### B3 — Engine endpoints with NO workspace proxy at all

16. **The tailored resume PDF cannot be downloaded** — [VALUE: high · EFFORT: S] — `GET /api/documents/variants/{variant_id}/download` (src/applicant/app/routers/documents.py:134) serves the compiled PDF artifact; `applicant_engine.py` has no method for it, no proxy, no JS → the user can approve a resume they can never save/open. Wiring: client method → streaming proxy in `applicant_documents_routes.py` → download icon on `documentLibrary.js` variant cards and Debug variants tab. Anchor: src/applicant/app/routers/documents.py:134.
17. **Campaign deletion is impossible from the product** — [VALUE: high · EFFORT: S] — `DELETE /api/campaigns/{campaign_id}` (src/applicant/app/routers/campaigns.py:95) exists and correctly routes through `DataLifecycleService` + `ErasureService.delete_campaign` (src/applicant/application/services/erasure_service.py:22) for a clean purge — no client method, no proxy, no UI. Anchor: src/applicant/app/routers/campaigns.py:95.
18. **Vault master-key rotation unreachable** — [VALUE: med · EFFORT: S] — `POST /api/credentials/rotate-key` (src/applicant/app/routers/credentials.py:127) re-encrypts every secret under a fresh key; no proxy in `applicant_vault_routes.py` (which stops at store/capture/account/tenants) → the flagship vault-hygiene action is env/curl-only. Wiring: vault proxy + "Rotate encryption key" button in `applicantVault.js`. Anchor: src/applicant/app/routers/credentials.py:127.
19. **Conversion preview PDF download dark** — [VALUE: med · EFFORT: S] — `GET /api/conversion/{campaign_id}/preview/download` (src/applicant/app/routers/conversion.py:72) serves the compiled LaTeX preview the user is asked to accept/reject; the accept decision (`applicantOnboarding.js`) is made without being able to open the actual PDF. 20. **Engine model endpoints can be added but never edited or removed** — [VALUE: med · EFFORT: S] — engine `PATCH /api/model-endpoints/{id}` (src/applicant/app/routers/model_endpoints.py:116) and `DELETE` (:126) have no proxy — `applicant_setup_routes.py:274-327` covers only list/add/test/models — so stale or mistyped endpoints accumulate forever in the ladder. Wiring: two thin proxies + edit/delete affordances in the existing endpoint manager. Anchor: src/applicant/app/routers/model_endpoints.py:116.
21. **Deferred long-essay generation dark** — [VALUE: med · EFFORT: M] — `POST /api/documents/deferred-essay` (src/applicant/app/routers/documents.py:265) generates the "come back to this essay question later" answer; no client/proxy/JS — the pre-fill stop-boundary can park an essay, but the user has no surface to trigger/review the deferred draft. 22. **Standalone redline render endpoint unproxied** — [VALUE: low · EFFORT: S] — `POST /api/documents/redline` (src/applicant/app/routers/documents.py:204) renders an arbitrary base-vs-new redline; the UI reaches redlines only via `/{id}/review` session state. 23. **Feedback is write-only — the user can never see what they taught it** — [VALUE: med · EFFORT: M] — `GET /api/feedback` is a stub index (src/applicant/app/routers/feedback.py:32) while `FeedbackSummaryProvider` (src/applicant/application/services/feedback_history.py:50) already walks per-application feedback for the curation nudge — the read-model exists, no endpoint returns it. 24. **MCP tool surface is entirely undocumented product capability** — [VALUE: low · EFFORT: M] — `/mcp/tools` list + `/mcp/tools/call` (src/applicant/app/routers/mcp.py:56,76) expose the engine to external agents, but the `api` service is internal-only and nothing in the front-door or docs mentions it — a differentiator (drive Applicant from your own agent) that no user can discover or reach.
